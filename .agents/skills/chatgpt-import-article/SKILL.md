# Импорт статьи из ChatGPT

## Когда использовать

- Пользователь хочет превратить экспорт чата из ChatGPT в статью для этого репозитория.
- Пользователь просит сделать выжимку, переработать диалог в материал уровня статьи, собрать конспект в `docs/`.
- Исходником является Markdown-экспорт в `raw/chatgpt/` или другой явно указанный экспорт, который нужно привести к такому workflow.

## Базовые принципы

- Пиши по-русски, если пользователь явно не просит другой язык.
- Финальный результат должен быть самостоятельной статьёй, а не видимой расшифровкой формата вопрос-ответ.
- Используй MyST-нотацию для публикуемых документов.
- Mermaid-диаграммы разрешены по умолчанию, если реально улучшают понимание.
- Для текущего репозитория базовая модель обработки - один проход по исходному Markdown-файлу нормального размера.
- Если тема явно расползается, сначала предложи пользователю разбиение на несколько статей.
- Дополнительный ресёрч допустим, но по умолчанию не обязателен. Если проверка нужна, опирайся на первоисточники.

## Ожидаемая структура

Используй следующие пути:

```text
raw/chatgpt/<chat-export>.md
work/chatgpt/<slug>/
docs/<slug>.md
index.md
```

Внутри `work/chatgpt/<slug>/` используй:

```text
outline.md
theses.md
decisions.md
meta.yaml
sections/
  01-<section>.md
  02-<section>.md
```

## Workflow

### 1. Подготовка

- Определи исходный файл.
- Если пользователь не указал файл явно, найди подходящий экспорт в `raw/chatgpt/` и уточни выбор только если есть неоднозначность.
- Сгенерируй стабильный `slug` для статьи.
- Создай `work/chatgpt/<slug>/meta.yaml` с минимумом полей:
  - `source`
  - `slug`
  - `status`
  - `target`

### 2. План статьи

- Прочитай исходный Markdown-экспорт.
- Сформируй `work/chatgpt/<slug>/outline.md`.
- Включи:
  - рабочее название;
  - структуру разделов;
  - что выбрасывается как шум, повторы или тупиковые ветки;
  - где уместны Mermaid-диаграммы;
  - нужен ли split.
- Зафиксируй важные решения в `decisions.md`, если по ходу появляются содержательные развилки.
- Покажи пользователю краткое summary плана и остановись до получения одобрения.

### 3. Тезисы по секциям

- После одобрения плана подготовь `work/chatgpt/<slug>/theses.md`.
- Для каждой секции зафиксируй 3-7 тезисов.
- Отдельно отметь:
  - тонкие различия, которые стоит подсветить;
  - что лучше не раздувать;
  - где нужна диаграмма вместо длинного объяснения.
- Покажи пользователю summary тезисов и дождись одобрения.

### 4. Черновики секций

- После одобрения тезисов пиши секции по одной в `work/chatgpt/<slug>/sections/`.
- Именуй файлы с числовым префиксом: `01-...md`, `02-...md`.
- После каждой секции показывай краткое summary и жди ревью пользователя, прежде чем двигаться дальше, если пользователь не попросил другой ритм.
- Если для секции нужна диаграмма, используй Mermaid. При сложной диаграмме отдельно подключи скилл `mermaid-diagrams`.

### 5. Финальная статья

- После утверждения всех секций собери финальный документ в `docs/<slug>.md`.
- Вычисти следы диалога, повторов и переходов, которые имеют смысл только внутри чата.
- Выравнивай стиль по всей статье уже после сборки, а не за счёт переписывания одобренных секций с нуля без необходимости.
- Если `index.md` уже существует, обнови его.
- Если `index.md` отсутствует, создай минимальное плоское оглавление с ссылкой на новую статью.
- Обнови `meta.yaml` со статусом финальной публикации.

## Область ответственности

- Итоговая статья публикуется как самостоятельный материал без следов исходного диалога.
- Workflow включает checkpoint-ревью: план, тезисы, черновики секций, финальная сборка.
- Скилл охватывает преобразование исходного экспорта в рабочие артефакты и финальную публикацию.
- Публикуемый слой остаётся плоским: `docs/<slug>.md` и `index.md`.

## Выходные артефакты

Минимальный ожидаемый набор:

- `work/chatgpt/<slug>/outline.md`
- `work/chatgpt/<slug>/theses.md`
- `work/chatgpt/<slug>/sections/*.md`
- `work/chatgpt/<slug>/meta.yaml`
- `docs/<slug>.md`
- `index.md` (созданный или обновлённый)
