# Дизайн воркфлоу импорта из ChatGPT

## Цель

Сделать практичный MVP-воркфлоу для превращения длинного диалога из ChatGPT в качественную статью в формате MyST в этом репозитории для подготовки к собеседованиям по темам вроде HTTP, WebSocket, TLS, atomic'ов и структур данных.

Воркфлоу должен:

- работать с локально сохранённым экспортом чата, а не со вставленной в диалог перепиской;
- выдавать самостоятельную статью, а не сохранять формат диалога;
- поддерживать пользовательское ревью на нескольких контрольных точках до написания финального текста;
- пока сохранять простую структуру репозитория;
- по умолчанию разрешать Mermaid-диаграммы там, где они реально улучшают понимание.

## Не-цели

- Полная автоматизация для всех возможных форматов экспорта.
- Обязательная внешняя перепроверка фактов для каждой статьи.
- Тематическая таксономия или глубокая иерархия каталогов в опубликованных документах.
- Связывание воркфлоу импорта контента с репозиторными хуками валидации MyST.

## Исходный материал

Для MVP принимаемым исходным форматом является локальный Markdown-экспорт диалога из ChatGPT, помещённый в `raw/chatgpt/`.

Ожидаемый путь к исходнику:

```text
raw/chatgpt/<chat-export>.md
```

Воркфлоу проектируется вокруг Markdown-экспортов, полученных вне агента, например через браузерное расширение. Экспорт считается источником истины для конкретного прогона импорта.

## Размер и модель обработки

Текущий размер чата, примерно 188 KB Markdown, следует считать близким к верхней границе для MVP.

Для такого класса входных данных импорт должен делать один проход по полному исходному файлу.

Если будущие чаты окажутся заметно больше, воркфлоу можно будет позже пересмотреть, но MVP должен оптимизироваться под текущую реальность, а не под преждевременно обобщённую систему импорта.

## Модель результата

Целевой режим по умолчанию:

- один чат -> одна статья

При этом агент может предложить разбиение на несколько статей, если беседа явно расползается на несколько крупных тем. Такое разбиение должно предлагаться явно, а не применяться молча.

Финальная опубликованная статья должна быть:

- самостоятельной технической статьёй;
- написанной в MyST-диалекте Markdown;
- без видимой структуры вопрос-ответ;
- с акцентами на скользких или легко понимаемых неправильно местах только там, где это реально помогает;
- не загнанной в жёсткий дидактический шаблон, если материал этого не требует.

## Структура репозитория

Публикуемый слой намеренно остаётся простым.

```text
index.md
raw/
  chatgpt/
    <original exports>
work/
  chatgpt/
    <slug>/
      outline.md
      theses.md
      decisions.md
      meta.yaml
      sections/
        01-<section>.md
        02-<section>.md
docs/
  <article>.md
```

### Правила структуры

- `index.md` лежит в корне репозитория и служит плоским оглавлением.
- `docs/` пока представляет собой плоский список опубликованных статей.
- `raw/chatgpt/` хранит исходные экспорты.
- `work/chatgpt/<slug>/` хранит только промежуточные артефакты, которые реально участвуют в ревью, а не произвольный технический шум.

## Воркфлоу ревью

MVP опирается на поэтапную ручную валидацию, а не на слепую генерацию полной статьи за один заход.

### Этап 1: План статьи

Агент читает полный экспорт чата и пишет `work/chatgpt/<slug>/outline.md`, содержащий:

- рабочее название статьи;
- предлагаемую структуру разделов;
- контент, который стоит исключить как шум или повтор;
- места, где помогут Mermaid-диаграммы;
- рекомендацию по разбиению на несколько статей, если она нужна.

Пользователь ревьюит и утверждает план статьи до перехода дальше.

### Этап 2: Тезисы по секциям

Агент пишет `work/chatgpt/<slug>/theses.md`, содержащий тезисы по секциям:

- 3-7 пунктов на секцию;
- ключевые различия и тонкие места, которые стоит подчеркнуть;
- места, где следует ограничить объём, чтобы статья не распухала.

Пользователь ревьюит и утверждает тезисы до написания текста секций.

### Этап 3: Черновики секций

Агент пишет секции по одной в:

```text
work/chatgpt/<slug>/sections/
```

Это позволяет независимо ревьюить тон, глубину и ход рассуждения по каждой секции. Если секция ушла не туда, переписывается только она.

### Этап 4: Финальная статья

После утверждения секций агент собирает финальную статью в:

```text
docs/<slug>.md
```

После этого он создаёт или обновляет `index.md`, добавляя плоскую ссылку на новую статью.

## Использование Mermaid

Mermaid включён по умолчанию там, где он действительно улучшает понимание.

Уместные сценарии использования:

- sequence diagrams для протокольных сценариев;
- state diagrams для состояний соединения или протокола;
- layered views для объяснения сетевых стеков;
- точечные сравнения там, где диаграмма яснее текста.

Mermaid не должен добавляться как декоративный элемент.

## Политика ресёрча

Основная цель MVP - переработка уже существующего чата в структурированную статью.

Дополнительный ресёрч допустим, но по умолчанию не требуется. Агент может выполнить точечную проверку или добор материала, если замечает пробел, который существенно влияет на ясность или корректность, но внешний поиск не является дефолтным шагом для каждой статьи.

## Граница ответственности скилла

Будущий скилл импорта должен описывать только содержательный workflow:

1. прочитать экспорт чата из `raw/chatgpt/`;
2. подготовить `outline.md`;
3. подготовить `theses.md`;
4. написать черновики секций;
5. собрать `docs/<slug>.md`;
6. обновить `index.md`.

Скилл должен инструктировать агента использовать MyST-нотацию в публикуемом результате.

Скилл ограничивается содержательным workflow импорта и подготовки статьи. Если автоматическая проверка MyST позже появится через hooks уровня harness или локальные команды, скилл можно будет расширить тогда, если это окажется полезным.

## Граница ответственности harness

Репозиторная валидация, например проверка MyST-разметки или форматные проверки, относится к уровню harness и живёт отдельно от workflow импорта.

Это сохраняет переиспользуемость процесса как для статей, импортированных из чатов, так и для статей, написанных с агентом с нуля.

## Рекомендация

Принять этот процесс как MVP для первой импортируемой статьи, используя текущий Markdown-экспорт как пилотный вход.

Не переусложнять автоматизацию заранее. Сначала валидировать процесс на одной реальной статье, а уже потом решать, нужна ли скиллу импорта и hooks-валидации дополнительная сложность.
