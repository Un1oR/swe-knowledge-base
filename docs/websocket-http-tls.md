# WebSocket, HTTP и TLS: путь от URL до фреймов

Эта статья собирает один сквозной маршрут: что происходит, когда приложение открывает `wss://example.com/feed`, как имя превращается в адрес, где начинается TLS, как HTTP переключается на WebSocket, как дальше устроены фреймы и что меняется при HTTP/2, HTTP/3 и QUIC.

Фокус здесь не на полном справочнике стандартов, а на рабочей инженерной модели. Важнее понимать границы слоёв: DNS ищет endpoint, TCP даёт byte stream, TLS защищает канал и аутентифицирует сервер, HTTP оформляет opening handshake, WebSocket задаёт собственное фреймирование и message-oriented контракт.

## Сквозной сценарий: от `wss://example.com/feed` до первых байтов

Удобнее всего держать сетевое взаимодействие не как набор разрозненных слов `DNS`, `TCP`, `TLS`, `HTTP` и `WebSocket`, а как последовательность слоёв, каждый из которых решает свою задачу.

Допустим, клиентское приложение хочет подключиться к:

```text
wss://example.com/feed
```

В грубой, но полезной модели путь выглядит так:

```{mermaid}
sequenceDiagram
    autonumber
    participant App as Приложение
    participant Resolver as libc/NSS resolver
    participant TCP as TCP endpoint
    participant TLS as TLS layer
    participant HTTP as HTTP/1.1 layer
    participant WS as WebSocket layer

    App->>Resolver: getaddrinfo("example.com", "443")
    Resolver-->>App: IP-адреса
    App->>TCP: socket() + connect(ip, 443)
    TCP-->>App: байтовый поток
    App->>TLS: TLS handshake
    TLS-->>App: защищённый канал
    App->>HTTP: GET /feed + Upgrade: websocket
    HTTP-->>App: 101 Switching Protocols
    App->>WS: WebSocket frames
```

Сначала приложение разбирает URL. Схема `wss://` говорит две важные вещи: нужен WebSocket и он должен идти внутри TLS. Если порт явно не указан, для такого подключения обычно используется 443, как и для `https://`. Для `ws://` аналогичная базовая схема не включает TLS и обычно использует порт 80.

Дальше hostname нужно превратить в адрес. Низкоуровневый `connect()` не подключается к строке `example.com`; он работает с `sockaddr`, то есть с конкретным IP-адресом и портом. Поэтому до TCP-соединения обычно вызывают `getaddrinfo()`, который возвращает набор подходящих адресов. Что именно происходит внутри резолвинга, зависит от libc, NSS, локальных конфигов и resolver daemon; для первого прохода важно только то, что на выходе у клиента появляются адреса, к которым можно пробовать `connect()`.

После этого создаётся TCP-сокет и устанавливается TCP-соединение. TCP не знает ничего ни про HTTP, ни про WebSocket. Он даёт упорядоченный байтовый поток между двумя endpoint'ами. В этом потоке нет границ сообщений: всё, что выше TCP, должно само понимать, где заканчивается заголовок HTTP, TLS record или WebSocket frame.

Так как схема `wss://`, следующим шагом поверх TCP запускается TLS handshake. Только после успешного TLS клиент получает защищённый канал, внутри которого можно отправлять прикладные байты. Поэтому HTTP-запрос WebSocket Upgrade для `wss://` физически едет уже внутри TLS, а не до него.

Классический WebSocket начинается как HTTP/1.1-запрос:

```http
GET /feed HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: ...
Sec-WebSocket-Version: 13
```

Если сервер согласен, он отвечает `101 Switching Protocols`. С этого момента HTTP-обмен завершён: дальше по тому же соединению идут уже WebSocket-фреймы. Отдельного второго WebSocket-рукопожатия после `101` нет.

Главная мысль этой карты: каждый слой добавляет свою гарантию и свою форму данных.

- DNS/resolver помогает найти адрес, но не доказывает личность сервера.
- TCP даёт надёжный упорядоченный поток байтов, но не даёт сообщений.
- TLS защищает канал и проверяет, что клиент говорит с тем именем, к которому подключался.
- HTTP/1.1 оформляет opening handshake WebSocket через `Upgrade`.
- WebSocket после `101` задаёт собственное фреймирование сообщений поверх уже установленного потока.

Если на интервью описывать подключение к `wss://example.com/feed`, достаточно начать с этой последовательности. Детали важны, но они должны цепляться за правильный порядок: DNS до TCP, TLS до HTTP Upgrade, WebSocket frames после успешного `101`.

## DNS и системный резолвинг: что реально делает `getaddrinfo`

В прикладном коде мы часто говорим: "подключаемся к `example.com`". Низкоуровнево это неточная формулировка. TCP-соединение устанавливается не с доменным именем, а с конкретным адресом и портом.

Системный вызов `connect()` принимает структуру адреса, например `sockaddr_in` для IPv4 или `sockaddr_in6` для IPv6. В ней уже должен быть IP-адрес. Поэтому перед `connect()` клиенту обычно нужно выполнить разрешение имени:

```text
example.com + 443 -> список sockaddr -> socket() + connect()
```

В низкоуровневом коде для этого обычно используют `getaddrinfo()`:

```c
getaddrinfo("example.com", "443", &hints, &result);
```

На выходе получается не один "правильный IP", а список подходящих адресов. Там могут быть IPv6 и IPv4 адреса, разные варианты socket type, разные ordering rules. Клиент дальше выбирает адреса из этого списка и пробует подключиться. Некоторые реализации делают это последовательно, некоторые используют стратегии вроде быстрого fallback между IPv6 и IPv4.

### Где здесь libc, NSS и resolver

`getaddrinfo()` не стоит воспринимать как один syscall. Это библиотечный API, за которым находится политика резолвинга конкретной системы.

Упрощённо путь такой:

```{mermaid}
flowchart TD
    App[Приложение] --> GAI[getaddrinfo]
    GAI --> NSS[libc / NSS policy]
    NSS --> Hosts["/etc/hosts"]
    NSS --> Resolver["DNS resolver: libc или локальный daemon"]
    Resolver --> Cache["локальный кеш"]
    Resolver --> ServerChoice["выбор DNS-сервера"]
    ServerChoice --> Static["resolv.conf / статическая настройка"]
    ServerChoice --> Dynamic["DHCP / VPN / split-DNS"]
    Static --> DNSServer["DNS-сервер"]
    Dynamic --> DNSServer
    DNSServer --> Query["запрос к DNS-серверу"]
    Query --> DNSAnswer["DNS-ответ: A/AAAA records"]
    DNSAnswer -.-> Cache
    Hosts --> Records["адресные записи"]
    Cache --> Records
    DNSAnswer --> Records
    Records --> Result["список sockaddr"]
    Result --> Connect["socket + connect"]
```

NSS означает Name Service Switch. В glibc-мире это механизм, который определяет, из каких источников и в каком порядке брать сведения об именах. Для хостов порядок обычно описывается в `/etc/nsswitch.conf`. Например, система может сначала смотреть локальные файлы, затем DNS, затем mDNS или другие источники.

Локальный файл `/etc/hosts` может переопределить имя без обращения к DNS. Поэтому если там написано:

```text
127.0.0.1 example.com
```

то для обычного системного резолвинга `example.com` может превратиться в loopback-адрес.

DNS-сервер, к которому нужно обращаться, тоже обычно выбирается не ядром. В классической схеме resolver читает `/etc/resolv.conf`, где перечислены `nameserver` и опции. В более динамичной схеме адрес DNS-сервера может прийти из DHCP, VPN-конфигурации или split-DNS политики. После выбора сервера сетевая часть становится общей: отправить DNS-запрос к выбранному DNS-серверу, обычно UDP на порт 53, иногда TCP.

В современных дистрибутивах часто есть локальный resolver daemon, например `systemd-resolved`. Тогда приложение или libc может обращаться не сразу к внешнему DNS-серверу, а к локальному stub вроде `127.0.0.53`. Локальный daemon может ответить из кеша или сам выбрать внешний DNS-сервер с учётом DHCP, VPN, split-DNS и других правил. На схеме это отражено так: локальный кеш и DNS-ответ от сервера сходятся в общие адресные записи, из которых `getaddrinfo()` собирает список `sockaddr`. Пунктир от DNS-ответа к кешу показывает побочный эффект: полученный ответ может сохраниться локально и ускорить следующие резолвы.

### Что делает ядро

Ядро не обязано понимать DNS как "систему имён". Оно участвует ниже:

- открывает сокеты;
- отправляет UDP/TCP-пакеты к DNS-серверу или локальному stub;
- применяет маршрутизацию;
- устанавливает TCP-соединение после того, как адрес уже выбран.

Поэтому фраза "резолвинг делает ОС" бытовая и не совсем вредная, но на собеседовании лучше уточнять границу: политика резолвинга обычно живёт в user space, а ядро предоставляет транспортные примитивы.

### Что важно для клиента

Первое практическое следствие: `getaddrinfo()` может быть дорогой и блокирующей операцией. Даже если часто ответ берётся из кеша или `/etc/hosts`, в худшем случае резолвинг может пойти во внешнюю сеть, упереться в таймауты DNS, VPN или локального resolver daemon. В latency-sensitive клиенте это нельзя небрежно делать на горячем event loop потоке.

Второе следствие: результат резолвинга не доказывает, что по найденному адресу находится нужный сервер. DNS помогает найти endpoint, но не отвечает на вопрос "это действительно `example.com`?". Если злоумышленник или ошибочная конфигурация вернули другой IP, TCP всё равно может успешно подключиться.

Проверка личности начинается позже, на TLS-слое. Клиент подключается к IP-адресу, но при проверке сертификата сравнивает имя из URL, например `example.com`, с именами в сертификате сервера. Именно поэтому hostname из исходного URL нельзя терять после DNS-резолвинга: он ещё понадобится для SNI и hostname verification.

### Как это формулировать на интервью

Хорошая короткая формулировка:

> `connect()` работает с IP-адресом и портом. Hostname сначала проходит через `getaddrinfo()`, а дальше libc/NSS и системная конфигурация решают, смотреть ли `/etc/hosts`, локальный resolver или DNS. Ядро здесь даёт сокеты и маршрутизацию, но не является главным местом политики DNS-резолвинга. DNS-ответ сам по себе не доказывает идентичность сервера; это проверяется позже через TLS-сертификат и hostname verification.

Эта модель достаточно точна для разработчика: она объясняет, почему резолвинг может блокировать, почему результатом может быть несколько адресов, почему `/etc/hosts` влияет на сетевое поведение и почему TLS всё равно проверяет доменное имя, хотя TCP уже подключился к конкретному IP.

## TLS как слой поверх TCP

После DNS и TCP у клиента уже есть соединение с конкретным IP-адресом и портом. Но это ещё не значит, что соединение безопасное. TCP даёт надёжный упорядоченный поток байтов, а TLS добавляет поверх этого потока другое свойство: защищённый канал между приложениями.

Для `wss://example.com/feed` порядок такой:

```text
getaddrinfo -> socket/connect -> TLS handshake -> HTTP Upgrade -> WebSocket frames
```

TLS не является частью TCP и не является частью WebSocket. Это отдельный слой между транспортом и прикладным протоколом. HTTPS, WSS, IMAPS, SMTPS, MQTT over TLS и защищённые подключения к базам данных используют одну и ту же идею: сначала есть транспорт, чаще TCP, затем поверх него выполняется TLS handshake, а уже потом приложение читает и пишет свои данные через TLS-слой.

В коде это важно понимать буквально. Если используется OpenSSL или похожая библиотека, приложение обычно само создаёт TCP-сокет, делает `connect()`, привязывает сокет к TLS-объекту и запускает handshake. После успешного handshake полезные данные нельзя продолжать отправлять обычным `send()` и читать обычным `recv()` как будто TLS не существует. Прикладные байты должны идти через API TLS-библиотеки, например через `SSL_write()` и `SSL_read()`, потому что именно этот слой шифрует, проверяет целостность и ведёт внутренние состояния протокола.

### Что именно даёт TLS

TLS обычно вспоминают как "шифрование", но для сетевой модели этого мало. У него есть три ключевые задачи.

Первая - конфиденциальность. Наблюдатель в сети не должен читать прикладной трафик: HTTP-заголовки, cookies, WebSocket payload, команды к базе данных или любые другие данные, которые идут внутри защищённого канала.

Вторая - целостность. Наблюдатель не должен незаметно менять байты на пути. Современный TLS защищает записи так, что подмена, удаление или повтор отдельных фрагментов трафика обнаруживается на криптографическом уровне.

Третья - аутентификация стороны. В обычном веб-сценарии клиент проверяет сервер: "я действительно установил защищённый канал с `example.com`, а не с кем-то, кто оказался по этому IP". Клиентская аутентификация сертификатом тоже существует, но для базовой модели HTTPS/WSS чаще достаточно серверной аутентификации.

Без аутентификации шифрование не решило бы главную проблему. Атакующий посередине мог бы установить одно защищённое соединение с клиентом, другое с настоящим сервером и пересылать данные между ними, читая и меняя содержимое. TLS защищает от такого сценария не только потому, что договаривается о ключах, а потому что привязывает этот обмен к идентичности сервера.

### Почему DNS и IP не доказывают личность сервера

DNS помогает найти endpoint, но не доказывает, что endpoint принадлежит нужному сервису. DNS-ответ может быть подменён, испорчен настройками, переопределён в `/etc/hosts`, изменён корпоративной или VPN-политикой. TCP после этого всё равно честно подключится к тому IP-адресу, который ему дали.

Поэтому TLS-проверка должна быть привязана не к факту "мы куда-то подключились", а к имени из исходного URL. Если клиент открывает:

```text
wss://example.com/feed
```

то после TCP-соединения он всё ещё должен помнить `example.com`. Это имя нужно минимум в двух местах:

- в SNI, чтобы сервер мог выбрать правильный сертификат;
- в hostname verification, чтобы клиент проверил, что сертификат действительно подходит для `example.com`.

Типичная ошибка - считать, что валидная цепочка сертификата сама по себе уже достаточна. Это не так. Представим, что клиент хотел подключиться к `a.example`, но атакующий подменил DNS и направил его на свой сервер `b.example`. У атакующего может быть совершенно валидный сертификат для `b.example`, выпущенный доверенным центром сертификации. Если клиент проверяет только цепочку доверия, но не проверяет имя хоста, он может принять "валидный сертификат, но не на тот домен". Именно это закрывает hostname verification.

На практике клиент проверяет, что имя из URL совпадает с именами в сертификате, обычно в `subjectAltName`. Проверка цепочки и проверка имени - разные шаги:

- цепочка отвечает на вопрос "можно ли доверять этому сертификату как выданному доверенной CA";
- hostname verification отвечает на вопрос "выдан ли этот сертификат именно для имени, к которому я подключался".

Отключать вторую проверку так же опасно, как отключать первую, просто атака выглядит менее очевидной.

### Сертификат не подписывает ответ

Ещё одна частая неточность: "сервер подписывает ответ сертификатом". Сертификат сам ничего не подписывает. Сертификат содержит публичный ключ сервера, имена, для которых он действителен, срок действия и подпись центра сертификации, который этот сертификат выпустил.

Во время TLS handshake сервер доказывает, что владеет приватным ключом, соответствующим публичному ключу из сертификата. Упрощённо:

1. сервер отправляет сертификат или цепочку сертификатов;
2. клиент проверяет цепочку доверия и имя хоста;
3. сервер подписывает данные handshake своим приватным ключом;
4. клиент проверяет подпись публичным ключом из сертификата.

Так клиент получает важную связку: "этот сервер владеет приватным ключом от сертификата, который доверенная CA выдала для `example.com`". Если атакующий просто скопирует чужой сертификат, этого недостаточно: без приватного ключа он не сможет пройти проверку владения ключом.

В современном TLS handshake также договаривается о ключах шифрования для сессии. Подробно механизм DH/ECDHE лучше разбирать отдельно: там важна идея, как стороны получают общий секрет по открытому каналу. Для этой секции достаточно зафиксировать границу ответственности: handshake одновременно договаривается о параметрах защищённого канала, проверяет идентичность сервера и подтверждает, что стороны видели согласованный transcript handshake.

```{mermaid}
sequenceDiagram
    autonumber
    participant Client
    participant Server
    participant Store as Trust store

    Client->>Server: ClientHello SNI ALPN
    Server->>Client: ServerHello
    Server->>Client: Certificate chain
    Client->>Store: Validate certificate chain
    Client->>Client: Check hostname
    Server->>Client: CertificateVerify
    Server->>Client: Finished
    Client->>Server: Finished
    Client->>Server: Encrypted app data
```

Стрелка к `Trust store` на схеме не означает, что клиент обязательно ходит в сеть к центру сертификации во время каждого handshake. Обычно клиент проверяет подписи цепочки локально, используя trust store системы, браузера или самой TLS-библиотеки. В некоторых конфигурациях дополнительно проверяется отзыв сертификата, но базовая проверка доверия не требует синхронного запроса к корневой CA.

### SNI и ALPN

SNI, Server Name Indication, нужен из-за виртуального хостинга. На одном IP-адресе и порту может жить много доменов с разными сертификатами. Сервер должен понять, какой сертификат показать, ещё до того, как клиент отправит HTTP-заголовок `Host`, потому что HTTP для HTTPS/WSS пойдёт уже после TLS handshake. Поэтому клиент передаёт нужный hostname в `ClientHello`.

Для `wss://example.com/feed` это тот же `example.com`, который был в URL и который потом будет использоваться при проверке сертификата. Если SNI не отправить, сервер может вернуть сертификат по умолчанию, и проверка имени законно упадёт.

ALPN, Application-Layer Protocol Negotiation, решает другую задачу: стороны договариваются, какой прикладной протокол будет идти поверх TLS. Например, клиент может предложить `h2` и `http/1.1`, а сервер выбрать один вариант. Для классического WebSocket через HTTP/1.1 Upgrade важно, что если по ALPN выбран `http/1.1`, дальше внутри TLS пойдёт обычный HTTP/1.1 запрос с `Upgrade: websocket`. Если выбран HTTP/2, модель WebSocket уже другая и потребует отдельного разговора про Extended CONNECT. Для HTTP/3 похожая роль ALPN остаётся, но handshake живёт уже внутри QUIC поверх UDP, а не в классическом TLS-over-TCP соединении.

SNI отвечает на вопрос "для какого имени нужен сертификат". ALPN отвечает на вопрос "какой протокол будет поверх TLS". Это разные расширения, и путать их не стоит.

### SSL как историческое имя

В современных разговорах слово SSL часто осталось как историческое имя. OpenSSL называется OpenSSL, API содержит типы и функции вроде `SSL_CTX` и `SSL_connect`, люди говорят "SSL-сертификат", но практически речь обычно идёт о TLS.

Для нормального современного клиента разумная ментальная модель такая: SSLv2 и SSLv3 небезопасны и не должны использоваться, рабочие версии - TLS 1.2 и TLS 1.3. В легаси-средах могут встречаться старые настройки, но "попробуем хоть SSLv3, лишь бы подключиться" - плохая политика: именно downgrade к старым версиям исторически был отдельным классом проблем.

На интервью достаточно спокойно развести термины: SSL - историческое название семейства и бытовой термин, TLS - актуальный протокол. Если в коде OpenSSL встречается `SSL_*`, это не значит, что приложение должно включать старый SSL как протокол.

### STARTTLS как контраст

Для HTTPS и обычно для WSS используется implicit TLS: клиент подключается к порту, где сервер сразу ждёт TLS `ClientHello`. Для `https://example.com` без явного порта это обычно 443; браузер не обязан сначала идти на 80 и "улучшать" соединение до HTTPS. Аналогично `wss://` обычно означает TCP на 443, затем TLS, затем HTTP Upgrade внутри TLS.

STARTTLS устроен иначе. Клиент сначала подключается к обычному plaintext-протоколу на существующем порту, обменивается начальными командами или capabilities, затем отправляет команду вроде `STARTTLS`, и со следующего байта стороны начинают TLS handshake в том же TCP-соединении.

Упрощённо:

```text
TCP connect -> plaintext protocol greeting -> STARTTLS -> TLS handshake -> protected protocol
```

Такой подход исторически полезен для протоколов и инфраструктур, где уже существовал один массовый порт и нужно было добавить TLS без полного разрыва совместимости. Типичный пример - почтовые протоколы, особенно SMTP. Минус тоже понятен: если TLS не является обязательным, появляется риск downgrade, когда атакующий или сломанный посредник мешает сторонам договориться о переходе на TLS. Поэтому implicit TLS проще как модель безопасности: защищённый режим начинается с первого прикладного байта протокола на этом порту.

Для маршрута `wss://` STARTTLS не нужен. Здесь полезно помнить только контраст: TLS может начинаться сразу после TCP, как в HTTPS/WSS, или включаться командой внутри уже начатого plaintext-протокола, как в STARTTLS.

### Как это формулировать на интервью

Короткая версия:

> TLS - это отдельный слой поверх TCP. TCP даёт поток байтов, а TLS поверх него делает handshake, проверяет сервер через цепочку сертификатов и hostname verification, договаривается о ключах и дальше защищает прикладные данные. DNS и IP не доказывают личность сервера, поэтому клиент должен проверять сертификат именно на hostname из URL. Сертификат не подписывает ответ: сервер доказывает владение приватным ключом, а клиент проверяет это публичным ключом из сертификата. SNI помогает серверу выбрать сертификат, ALPN выбирает протокол поверх TLS. SSL сегодня в основном историческое название; практически думаем про TLS 1.2 и 1.3.

Для разработчика это ещё и практическая граница API: сначала обычный сокет и `connect()`, затем TLS-библиотека и её handshake, затем чтение и запись прикладных данных через TLS-слой, а не напрямую через TCP-сокет.

## Как стороны договариваются о секрете

В предыдущей секции TLS выглядел как слой, который после TCP делает handshake, проверяет сервер и превращает обычный байтовый поток в защищённый канал. Теперь полезно разобрать центральный вопрос: как клиент и сервер получают общий секрет, если все сообщения handshake могут наблюдаться в сети.

Наивная идея "просто отправить ключ серверу" не подходит: наблюдатель увидит этот ключ. Поэтому современный TLS обычно использует не передачу готового секрета, а key agreement: обе стороны обмениваются публичными значениями и независимо вычисляют один и тот же общий секрет. Классическая модель для понимания - Diffie-Hellman.

### Классический Diffie-Hellman

В DH есть публичные параметры, которые не являются секретом:

- большое простое число $p$;
- генератор $g$, подходящий для выбранной группы.

Их может видеть кто угодно. Дальше стороны делают следующее.

Клиент выбирает приватное случайное число $a$. Это число не отправляется по сети. Вместо него клиент вычисляет публичное значение:

$$
A = g^a \bmod p
$$

Сервер выбирает своё приватное случайное число $b$ и публикует:

$$
B = g^b \bmod p
$$

После обмена публичными значениями клиент вычисляет:

$$
S = B^a \bmod p = (g^b)^a \bmod p = g^{ab} \bmod p
$$

Сервер вычисляет:

$$
S = A^b \bmod p = (g^a)^b \bmod p = g^{ab} \bmod p
$$

> **Почему это не "вынести модуль из-под степени"**
>
> Здесь важно не прочитать формулу как "модуль можно просто вынести из-под степени". Более точная идея такая: **если два числа дают один и тот же остаток по модулю $p$, то их степени тоже дают один и тот же остаток по модулю $p$**.
>
> Пусть
>
> $$ x \equiv y \pmod p $$
>
> По определению это значит, что разность $x - y$ делится на $p$ без остатка:
>
> $$ p \mid (x - y) $$
>
> Для целого $n \ge 1$ смотрим уже на разность $x^n - y^n$. Она раскладывается так:
>
> $$ x^n - y^n = (x - y)(x^{n-1} + x^{n-2}y + \dots + xy^{n-2} + y^{n-1}) $$
>
> Первый сомножитель делится на $p$, значит и всё произведение делится на $p$. Следовательно:
>
> $$ p \mid (x^n - y^n) $$
>
>а это и означает:
>
> $$ x^n \equiv y^n \pmod p $$
>
> В нашем случае
>
> $$ B = g^b \bmod p $$
>
> Это значит не "$B$ буквально равно $g^b$", а:
>
> $$ B \equiv g^b \pmod p $$
>
> То есть $B$ — это остаток от $g^b$ по модулю $p$. Они могут быть разными как обычные числа, но дают один и тот же остаток по модулю $p$.
>
> Дальше клиент возводит публичное $B$ в свою приватную степень $a$:
>
> $$ B^a \bmod p $$
>
> И тут применяется свойство:
>
> $$ B \equiv g^b \pmod p $$
>
> значит:
>
> $$ B^a \equiv (g^b)^a \pmod p $$
>
> то есть:
>
> $$ B^a \bmod p = (g^b)^a \bmod p $$

Обе стороны пришли к одному и тому же $S$, хотя приватные $a$ и $b$ не покидали свои машины. Интуитивный трюк здесь простой: степени можно "доприменить" в разном порядке, а результат будет тем же, потому что в показателе получается $ab$.

Наблюдатель видит $p$, $g$, $A$ и $B$. Но чтобы вычислить тот же секрет напрямую, ему нужно восстановить $a$ из $A = g^a \bmod p$ или $b$ из $B = g^b \bmod p$. Это задача дискретного логарифма. Для правильно выбранных параметров она считается вычислительно неподъёмной.

Важно не смешивать это с RSA. Факторизация больших чисел - это RSA-мир. DH держится на другой трудной задаче: дискретном логарифме в выбранной группе.

```{mermaid}
flowchart LR
    Client[Client] --> ClientSecret[Private a]
    Server[Server] --> ServerSecret[Private b]
    Public[Public parameters p and g] --> Client
    Public --> Server
    ClientSecret --> A[Public A]
    ServerSecret --> B[Public B]
    A --> Network[Open network]
    B --> Network
    Network --> ClientCalc[Client computes shared secret]
    Network --> ServerCalc[Server computes shared secret]
    ClientSecret --> ClientCalc
    ServerSecret --> ServerCalc
    ClientCalc --> Same[Same session secret]
    ServerCalc --> Same
    Network --> Observer[Observer sees public values only]
```

Эта диаграмма намеренно не показывает все детали реального TLS. Для интервью важнее ментальная модель: приватные числа остаются локальными, публичные значения можно отправлять по открытому каналу, общий секрет появляется с двух сторон, а наблюдатель упирается в дискретный логарифм.

### ECDHE: та же идея на эллиптических кривых

ECDHE устроен похожим образом, но вместо возведения числа в степень по модулю используется операция со скалярами и точками на эллиптической кривой.

Публично известны параметры группы и базовая точка. Клиент выбирает приватный скаляр `a` и публикует соответствующую публичную точку. Сервер выбирает приватный скаляр `b` и публикует свою публичную точку. Затем клиент применяет свой приватный скаляр к публичной точке сервера, а сервер применяет свой приватный скаляр к публичной точке клиента. В результате обе стороны получают один и тот же общий секрет.

Для атакующего задача аналогична по смыслу: по базовой точке и публичной точке практически невозможно восстановить приватный скаляр, если выбрана нормальная современная группа. Здесь не нужно выводить формулы кривых или доказывать стойкость. Достаточно понимать перенос структуры:

- в классическом DH есть $p$, $g$, приватные $a$/$b$ и публичные $A$/$B$;
- в ECDHE есть публичная группа с базовой точкой, приватные скаляры и публичные точки;
- в обоих случаях стороны договариваются о секрете, не передавая сам секрет.

На практике TLS сегодня чаще использует именно ECDHE-подобные группы, потому что они дают компактные публичные значения и быстрые вычисления при хорошем уровне стойкости.

### Что означает `E` и зачем нужна forward secrecy

Буква `E` в `DHE` и `ECDHE` означает ephemeral: временный, одноразовый. Для каждой сессии стороны генерируют новые приватные значения `a` и `b`. Они не являются долгосрочным ключом сервера и не должны переиспользоваться как постоянный секрет.

Это даёт forward secrecy. Если через год у сервера утечёт приватный ключ сертификата, записанный сегодня трафик не должен автоматически расшифроваться. Старые сессионные ключи были выведены из одноразовых DH/ECDHE-секретов, а не просто зашифрованы долгосрочным ключом сервера. Поэтому одного будущего компрометационного события недостаточно, чтобы прочитать прошлые записи соединений.

Forward secrecy не означает, что можно не заботиться о приватном ключе сертификата или о реализации TLS. Она означает более узкую, но очень важную гарантию: компрометация долгосрочного ключа не раскрывает автоматически старые сессионные ключи.

### Зачем тогда сертификат

DH сам по себе отвечает на вопрос "как получить общий секрет по открытому каналу". Он не отвечает на вопрос "с кем именно я этот секрет получил".

Если убрать аутентификацию, атакующий посередине может провести один DH-обмен с клиентом, другой DH-обмен с настоящим сервером и стать переводчиком между двумя защищёнными каналами. Клиент будет думать, что у него общий секрет с сервером, сервер будет думать, что у него общий секрет с клиентом, а посредник будет читать и менять данные между ними.

Сертификат сервера нужен не для математического вычисления DH-секрета. Он нужен, чтобы привязать handshake к идентичности сервера:

- клиент проверяет цепочку доверия сертификата;
- клиент проверяет hostname из URL;
- сервер доказывает владение приватным ключом, соответствующим публичному ключу из сертификата;
- handshake связывает публичные параметры обмена и transcript с этой аутентификацией.

`Transcript` здесь - это не текстовая расшифровка, а накопленная история handshake-сообщений в том порядке, в котором их видит сторона. На практике реализация обычно ведёт хеш transcript'а, а не хранит одну большую строку. В него попадают сообщения вроде `ClientHello`, `ServerHello`, выбранные параметры, расширения, ECDHE `key_share`, сертификат и другие handshake-данные.

В TLS 1.3 сервер доказывает владение приватным ключом через сообщение `CertificateVerify`. Упрощённо оно выглядит так:

```text
signature = Sign(server_certificate_private_key, hash(handshake_transcript))
```

Клиент строит свой transcript, берёт публичный ключ из сертификата и проверяет подпись:

```text
Verify(server_certificate_public_key, signature, hash(handshake_transcript))
```

Реальная подпись в TLS 1.3 включает ещё специальную контекстную строку, чтобы подпись нельзя было переиспользовать в другом протоколе или другом месте handshake. Но главная идея именно такая: сервер подписывает не абстрактное "я настоящий", а текущую историю handshake. Поэтому если MITM подменит ECDHE `key_share`, выбранные параметры или другие handshake-сообщения, transcript у клиента изменится и подпись перестанет проверяться.

Поэтому правильная формулировка такая: ECDHE договаривается о секрете, а сертификат и подпись защищают этот обмен от MITM.

### RSA key exchange и `ECDHE_RSA`

RSA в TLS может означать разные вещи, и это частый источник путаницы.

В старом TLS, особенно в TLS 1.2 и раньше, существовал RSA key exchange. Упрощённо он работал так:

1. сервер присылал сертификат с RSA-публичным ключом;
2. клиент проверял сертификат и hostname;
3. клиент генерировал `pre_master_secret`;
4. клиент шифровал `pre_master_secret` RSA-публичным ключом сервера и отправлял его серверу;
5. сервер расшифровывал это значение своим RSA-приватным ключом;
6. обе стороны выводили из `pre_master_secret` сессионные ключи.

Это не DH. Здесь клиент фактически переносит зародыш секрета серверу, зашифровав его публичным ключом сервера. Главный минус такой модели - отсутствие forward secrecy: если приватный RSA-ключ сервера позже утечёт, старые записанные handshake'и могут стать расшифровываемыми.

Название вроде `ECDHE_RSA` означает другое. В такой схеме:

- `ECDHE` отвечает за key agreement, то есть за получение общего секрета;
- `RSA` отвечает за аутентификацию, то есть за подпись и доказательство владения ключом из сертификата.

В TLS 1.2 это делалось через `ServerKeyExchange`: сервер отправлял временные ECDHE-параметры и подписывал их ключом из сертификата. Подпись покрывала не только сами параметры, но и `client_random`/`server_random`, чтобы старую подпись нельзя было просто переиграть в другом handshake.

RSA в `ECDHE_RSA` не шифрует WebSocket payload и не переносит pre-master secret. Прикладные данные после handshake шифруются симметричными ключами, выведенными из секрета, согласованного через ECDHE (например, для `AES-GCM` или `ChaCha20-Poly1305`).

В TLS 1.3 статический RSA key exchange убрали из нормального пути, а старые названия cipher suite вроде `ECDHE_RSA` больше не описывают весь набор механизмов. Современная модель - DH-подобный key agreement для секрета и отдельная аутентификация сервера через сертификат и `CertificateVerify`, то есть подпись над handshake transcript.

### Как это формулировать на интервью

Короткая версия:

> В TLS стороны обычно не передают готовый ключ, а договариваются о секрете через ECDHE. В классическом DH это видно на модели $p$, $g$, приватных $a$ и $b$, публичных $A = g^a \bmod p$ и $B = g^b \bmod p$: обе стороны получают $g^{ab} \bmod p$, а наблюдатель видит только публичные значения и упирается в дискретный логарифм. ECDHE переносит ту же структуру на эллиптические кривые. Ephemeral-ключи дают forward secrecy. Сертификат не нужен для вычисления DH-секрета, но нужен для аутентификации сервера и защиты от MITM. В старом RSA key exchange клиент шифровал pre-master secret RSA-ключом сервера, а в `ECDHE_RSA` ECDHE делает key agreement, RSA только аутентифицирует сервер.

## WebSocket opening handshake

Классический WebSocket не начинается с отдельного бинарного протокола. Сначала клиент отправляет обычный HTTP/1.1-запрос, который просит сервер переключить это соединение с HTTP-семантики на WebSocket. Этот обмен и называется opening handshake.

Для `ws://example.com/feed` порядок такой:

```text
TCP connect -> HTTP/1.1 Upgrade -> WebSocket frames
```

Для `wss://example.com/feed` перед этим появляется TLS:

```text
TCP connect -> TLS handshake -> HTTP/1.1 Upgrade inside TLS -> WebSocket frames
```

То есть при `wss://` HTTP-запрос с `Upgrade: websocket` не идёт "рядом" с TLS и не предшествует ему. Он передаётся как прикладные данные уже внутри установленного защищённого TLS-канала.

### Минимальный запрос

Упрощённый opening handshake выглядит так:

```http
GET /feed HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://example.com
```

Это именно `GET`, потому что классический WebSocket поверх HTTP/1.1 стартует как запрос на ресурс, но с просьбой сменить протокол для уже открытого соединения.

`Host` указывает HTTP-серверу, к какому virtual host относится запрос. На одном IP-адресе и порту может жить много сайтов, поэтому имя хоста всё ещё важно, хотя TCP уже подключился к конкретному адресу. Для `wss://` похожее имя обычно уже использовалось раньше в SNI и hostname verification на TLS-слое, но HTTP-заголовок `Host` всё равно остаётся частью HTTP/1.1-запроса.

`Upgrade: websocket` говорит, что клиент хочет перейти с HTTP/1.1 на протокол WebSocket. Сам по себе этот заголовок недостаточен: HTTP требует ещё `Connection: Upgrade`, чтобы явно отметить, что `Upgrade` относится к текущему соединению, а не должен быть бездумно переслан как end-to-end заголовок через посредников.

`Sec-WebSocket-Version: 13` фиксирует версию протокола WebSocket, которая используется современными реализациями по RFC 6455. Если сервер не поддерживает эту версию, он должен отказать в handshake, а не переходить к фреймам по своим правилам.

`Sec-WebSocket-Key` - это клиентский nonce: случайные 16 байт, закодированные в base64. Его задача не в шифровании и не в аутентификации пользователя. Он нужен, чтобы сервер доказал, что понял именно WebSocket opening handshake и сформировал корректный WebSocket-ответ, а не случайно вернул обычную HTTP-страницу.

`Origin` часто отправляют браузеры. Он помогает серверу принять решение, разрешать ли WebSocket-подключение со страницы данного origin. Это важный защитный механизм для браузерного сценария, но его не стоит смешивать с CORS как полноценной темой: для текущей секции достаточно помнить, что сервер может проверять `Origin` и отклонять неподходящие подключения.

Приложение также может добавить `Sec-WebSocket-Protocol`, если хочет договориться о subprotocol:

```http
Sec-WebSocket-Protocol: chat, market-data.v2
```

Это не версия WebSocket wire-format. Это прикладной протокол поверх WebSocket: например, сервер может выбрать `market-data.v2`, и тогда обе стороны будут понимать payload сообщений по этому прикладному контракту. Подробная политика выбора subprotocol не нужна для базовой картины handshake.

### Успешный ответ сервера

Если сервер принимает upgrade, он отвечает статусом `101 Switching Protocols`:

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

Если в запросе был `Sec-WebSocket-Protocol` и сервер выбрал один вариант, он указывает его в ответе:

```http
Sec-WebSocket-Protocol: market-data.v2
```

`101 Switching Protocols` - ключевой сигнал. Он означает: HTTP-сервер согласился переключить это соединение на другой протокол. Заголовки `Upgrade` и `Connection` подтверждают, что переключение именно на WebSocket, а `Sec-WebSocket-Accept` связывает ответ с клиентским `Sec-WebSocket-Key`.

После пустой строки, которая завершает HTTP-заголовки ответа, отдельного второго WebSocket-рукопожатия нет. Дальше по тому же TCP-соединению, а для `wss://` по тому же TLS-каналу, идут WebSocket-фреймы.

```{mermaid}
sequenceDiagram
    autonumber
    participant Client
    participant Server

    Client->>Server: HTTP/1.1 GET + Upgrade: websocket
    Server-->>Client: 101 Switching Protocols
    Client->>Server: WebSocket frames
    Server->>Client: WebSocket frames
```

На схеме важно не количество стрелок, а граница: до `101` стороны говорят HTTP/1.1, после `101` они уже интерпретируют байты как WebSocket-фреймы.

### Как считается `Sec-WebSocket-Accept`

Сервер берёт значение `Sec-WebSocket-Key` из запроса, конкатенирует с фиксированным GUID:

```text
258EAFA5-E914-47DA-95CA-C5AB0DC85B11
```

Важно: для этой операции берётся текстовое значение заголовка как оно пришло, а не base64-декодированные 16 байт nonce. Затем сервер считает SHA-1 от получившейся строки байт и кодирует результат в base64:

```text
Sec-WebSocket-Accept = base64(
    SHA-1(Sec-WebSocket-Key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11")
)
```

Для ключа из примера:

```http
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
```

получается:

```http
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

Здесь легко сделать неверный вывод из-за слов `Sec` и `SHA-1`. Это не криптографическая аутентификация сервера и не замена TLS. У клиента нет секрета, известного только настоящему серверу; `Sec-WebSocket-Key` отправляется открыто в HTTP-заголовке, а GUID фиксирован стандартом. Проверка нужна для другого: убедиться, что endpoint понял WebSocket handshake и не является обычным HTTP endpoint'ом, который случайно вернул похожий ответ.

Для `wss://` настоящую защиту канала, проверку имени сервера и защиту от MITM даёт TLS, разобранный в предыдущих секциях. `Sec-WebSocket-Accept` живёт уровнем выше и проверяет корректность WebSocket-переключения внутри уже выбранного HTTP-механизма.

### Что если ответ не `101`

Если сервер вернул `200 OK`, HTML-страницу, `404 Not Found`, `400 Bad Request`, редирект или любой другой не-`101` ответ, opening handshake не состоялся. Корректный клиент в этой ситуации не должен начинать парсить тело ответа как WebSocket-фреймы.

Это важная граница между секциями. Ошибка upgrade - ещё HTTP-ошибка. WebSocket wire-format начинается только после успешного `101 Switching Protocols` с корректными заголовками. Если клиент проигнорирует статус и начнёт трактовать байты вроде `HTTP/1.1 200 OK` или `<html>` как WebSocket frame, он уже находится в состоянии протокольной ошибки, а не в нормальном сценарии закрытия WebSocket.

### Как это формулировать на интервью

Короткая версия:

> Классический WebSocket opening handshake - это HTTP/1.1 `GET` с `Upgrade: websocket`, `Connection: Upgrade`, `Host`, `Sec-WebSocket-Key` и `Sec-WebSocket-Version: 13`. Сервер при успехе отвечает `101 Switching Protocols`, `Upgrade`, `Connection` и `Sec-WebSocket-Accept`, который считается как base64 от SHA-1 клиентского ключа плюс фиксированный GUID. Это не аутентификация, а проверка корректного WS-handshake. Для `wss://` весь HTTP Upgrade идёт внутри уже установленного TLS-канала. После `101` нет второго рукопожатия: по тому же соединению сразу идут WebSocket-фреймы. Если ответ не `101`, клиент не переходит к парсингу WebSocket frames.

Дальше начинается отдельная тема: как именно устроены WebSocket-фреймы, где в них длина, opcode, маска, фрагментация и границы сообщений поверх TCP-потока.

## WebSocket frames: границы, маскирование и фрагментация

В предыдущей секции важная граница была такой: до успешного `101 Switching Protocols` стороны говорят HTTP/1.1, после `101` HTTP-семантика для этого соединения закончилась. Дальше те же байты, которые приложение читает из TCP-сокета или из TLS-слоя при `wss://`, нужно интерпретировать уже как WebSocket frames.

Это не означает, что `read()` или `SSL_read()` начнут возвращать "по одному WebSocket-фрейму". TCP даёт упорядоченный поток байтов, а TLS поверх TCP сохраняет ту же прикладную модель: приложение получает куски байтов, но не получает границы сообщений WebSocket бесплатно.

Один вызов чтения может вернуть:

- только первый байт заголовка фрейма;
- два байта базового заголовка, но без extended length;
- заголовок и часть payload;
- хвост payload предыдущего фрейма и начало следующего;
- несколько маленьких фреймов подряд.

Поэтому WebSocket-реализация всегда содержит инкрементальный парсер. Он копит байты, восстанавливает границы фреймов по заголовку и длине payload, проверяет правила протокола, снимает маску при необходимости и только потом отдаёт приложению фреймы или уже собранные сообщения.

### Формат фрейма на wire

У WebSocket-фрейма нет завершающей последовательности байтов вроде `\r\n\r\n`. Конец фрейма определяется длиной, записанной в заголовке.

Минимальный заголовок занимает 2 байта.

Первый байт:

- `FIN` - последний ли это фрейм сообщения;
- `RSV1`, `RSV2`, `RSV3` - зарезервированные биты;
- `opcode` - тип фрейма.

В обычном соединении без negotiated extensions биты `RSV1/2/3` должны быть нулевыми. Если стороны договорились о расширении, например о сжатии, расширение может использовать один из этих битов. Без такой договорённости ненулевой `RSV` - протокольная ошибка.

Основные `opcode`:

- `0x0` - continuation frame;
- `0x1` - text frame;
- `0x2` - binary frame;
- `0x8` - close;
- `0x9` - ping;
- `0xA` - pong.

Close, ping и pong - control frames. Их правила, payload и политика закрытия соединения будут отдельной темой следующей секции. Сейчас важно только то, что эти opcode существуют и могут встречаться в общем потоке фреймов.

Второй байт:

- `MASK` - есть ли masking key;
- 7-битная базовая длина payload.

Длина кодируется так:

- `0..125` - это сама длина payload в байтах;
- `126` - дальше идут 2 байта extended length, то есть 16-bit длина;
- `127` - дальше идут 8 байт extended length, то есть 64-bit длина.

Extended length записывается в сетевом порядке байтов. В инженерной реализации на этом месте обычно появляются проверки максимального размера фрейма и сообщения: стандартный wire-format может выразить очень большую длину, но сервер или клиент не обязан принимать бесконечно большие payload.

После длины идёт masking key, если `MASK=1`. Это ровно 4 байта. После него начинается payload указанной длины.

В виде схемы порядок полей такой:

```text
byte 0: FIN RSV1 RSV2 RSV3 opcode
byte 1: MASK payload_len_7bit
maybe: 16-bit or 64-bit extended length
maybe: 4-byte masking key
payload bytes
```

Из этого следует главный инвариант парсера: пока заголовок не собран полностью, нельзя знать, где начинается payload; пока declared payload length не дочитана, нельзя перейти к следующему фрейму.

### Маскирование

Клиентские фреймы в WebSocket обязаны быть masked. Это требование действует и для `ws://`, и для `wss://`. TLS не отменяет маскирование, потому что маска - правило WebSocket-уровня, а не свойство нижнего транспорта.

Серверные фреймы обычно не masked. В классическом направлении "клиент -> сервер" сервер ожидает `MASK=1`; в направлении "сервер -> клиент" клиент обычно видит `MASK=0`.

Маскирование не является шифрованием. Оно не защищает данные от чтения и не заменяет TLS. Алгоритм намеренно простой: payload XOR'ится с 4-байтовым ключом по циклу.

```text
decoded[i] = encoded[i] XOR masking_key[i % 4]
```

Ключ передаётся рядом с фреймом, поэтому любой, кто видит фрейм, может снять маску. Смысл маскирования в другом: клиентские WebSocket-данные не должны ехать по сети как полностью контролируемая приложением повторяемая последовательность байтов.

Историческая модель риска такая: браузер выполняет JavaScript, которому злоумышленник может дать команду открыть `ws://`-соединение к контролируемому серверу. Если клиент отправляет payload как есть, скрипт может подобрать байты так, чтобы внутри WebSocket-потока они выглядели как обычный HTTP-запрос, например `GET /some-script.js HTTP/1.1`. Сломанный proxy, кеш или другой HTTP-посредник, который некорректно обрабатывает `Upgrade`, может пропустить WebSocket-framing и начать интерпретировать payload как HTTP. В худшем случае это превращается в cache poisoning: посредник кеширует ответ атакующего как будто это ответ на обычный HTTP-запрос.

Маскирование ломает именно этот контроль над wire-байтами: приложение выбирает логический payload, но браузер применяет непредсказуемый masking key для каждого фрейма, и на проводе получается другая последовательность байтов. Для `wss://` TLS в нормальном пути уже скрывает байты от HTTP-посредников, но требование `MASK=1` всё равно остаётся частью WebSocket-протокола. Для конфиденциальности и защиты от MITM нужен `wss://`, то есть TLS ниже WebSocket.

Для реализации это практическое правило: свой клиент должен ставить `MASK=1`, генерировать fresh 4-byte masking key для фрейма и XOR'ить payload; свой сервер должен проверять маску у входящих клиентских фреймов и снимать её до передачи payload выше.

### Message и frame

WebSocket даёт приложению message-oriented модель, но на wire передаёт frames. Это разные уровни.

Простое сообщение может поместиться в один фрейм:

```text
FIN=1, opcode=0x1, payload="text message"
```

или:

```text
FIN=1, opcode=0x2, payload=<binary bytes>
```

Но сообщение может быть фрагментировано. Тогда первый фрейм задаёт тип сообщения, а следующие фреймы продолжают его:

```text
FIN=0, opcode=0x1, payload=part1
FIN=0, opcode=0x0, payload=part2
FIN=1, opcode=0x0, payload=part3
```

Здесь всё вместе образует одно text message. Для binary message схема такая же, только первый opcode будет `0x2`. Continuation frames с `opcode=0x0` не начинают новое сообщение; они относятся к текущему незавершённому фрагментированному сообщению.

Важно, что внутри одного классического WebSocket-соединения нет встроенного `message_id` и нет мультиплексирования параллельных сообщений. Протокол опирается на упорядоченный поток фреймов: если началось фрагментированное data message, то до его завершения нельзя начать новое text или binary message. Поэтому continuation после вклинившегося control frame однозначно относится к тому же текущему сообщению.

Разрешённая картина:

```text
text FIN=0
ping
continuation FIN=0
pong
continuation FIN=1
```

Неразрешённая картина:

```text
text FIN=0
binary FIN=1
continuation FIN=1
```

Вторая последовательность пытается начать новое data message до завершения предыдущего фрагментированного сообщения. На уровне WebSocket это протокольная ошибка.

Если приложению нужны логические параллельные запросы и ответы, WebSocket сам их не сопоставит. Обычно это делают в payload:

```json
{"type":"quote.request","id":"42","symbol":"AAPL"}
```

или добавляют поля вроде `stream_id`, `request_id`, `type`, `method`, `channel`. Другой вариант - открыть несколько WebSocket-соединений. Классический WebSocket поверх HTTP/1.1 Upgrade не даёт "несколько WebSocket-каналов в одном TCP-соединении" как встроенную возможность.

### Зачем фрагментация, если есть большие длины

Wire-format действительно умеет выразить большой payload через extended length. Это не делает фрагментацию бесполезной.

Большой фрейм можно читать кусками после того, как заголовок уже разобран. Реализация не обязана одним махом аллоцировать буфер на весь declared length, если её API позволяет streaming. Но пока текущий фрейм не дочитан до declared length, следующий фрейм начать парсить невозможно: парсер находится внутри payload текущего фрейма.

Фрагментация даёт протокольные точки между фреймами. Именно туда могут вклиниваться control frames. Это особенно важно для больших или медленно производимых сообщений: peer может отправить ping, pong или close между фрагментами, не ожидая конца одного гигантского payload. Один огромный data frame такой точки не создаёт, даже если приложение внутри себя читает его payload чанками.

Практически фрагментация также помогает держать понятные лимиты: максимальный размер фрейма, максимальный размер сообщения, объём буферов на соединение, таймауты чтения. Хорошая библиотека может скрывать это за message API, но внутри ей всё равно приходится вести состояние: есть ли незавершённое сообщение, какого оно типа, сколько payload уже принято, можно ли сейчас принять continuation и можно ли прямо сейчас обработать control frame.

### State machine парсера

Инкрементальный WebSocket-парсер удобно мыслить как небольшую state machine:

```{mermaid}
stateDiagram-v2
    [*] --> ReadBaseHeader
    ReadBaseHeader --> NeedMore: less than 2 bytes
    NeedMore --> ReadBaseHeader
    ReadBaseHeader --> ResolveLength: base header ready
    ResolveLength --> ReadExtLen: len is 126 or 127
    ResolveLength --> ResolveMask: len is 0 to 125
    ReadExtLen --> NeedMoreExt: not enough bytes
    NeedMoreExt --> ReadExtLen
    ReadExtLen --> ResolveMask: length ready
    ResolveMask --> ReadMask: mask bit set
    ResolveMask --> ReadPayload: no mask
    ReadMask --> NeedMoreMask: not enough bytes
    NeedMoreMask --> ReadMask
    ReadMask --> ReadPayload: mask ready
    ReadPayload --> NeedMorePayload: payload incomplete
    NeedMorePayload --> ReadPayload
    ReadPayload --> ValidateFrame: payload complete
    ValidateFrame --> ProtocolError: invalid bits or order
    ValidateFrame --> Unmask: masked payload
    ValidateFrame --> Emit: unmasked payload
    Unmask --> Emit
    Emit --> ReadBaseHeader: next frame
    ProtocolError --> [*]
```

`Emit` здесь может означать разные вещи в зависимости от уровня API. Низкоуровневый парсер может отдать "получен frame". Более удобная WebSocket-библиотека обычно собирает continuation-фреймы и отдаёт приложению уже "получено text message" или "получено binary message". Для больших payload возможен streaming API, где приложение получает части сообщения по мере чтения, но правила wire-format от этого не исчезают.

### Как это формулировать на интервью

Короткая версия:

> После HTTP `101` байты соединения парсятся как WebSocket frames. TCP и TLS дают поток байтов, поэтому один `read` не соответствует одному фрейму: нужен инкрементальный парсер с буфером. WS-фрейм начинается с `FIN/RSV/opcode`, затем идут `MASK` и длина: `0..125` прямо в заголовке, `126` означает 16-bit extended length, `127` - 64-bit extended length; если `MASK=1`, дальше 4-byte masking key, потом payload. Клиентские фреймы обязаны быть masked даже внутри `wss://`; это XOR с 4-байтовым ключом, не шифрование и не замена TLS. Message и frame - разные уровни: сообщение может быть одним фреймом или цепочкой `text/binary FIN=0` плюс continuation frames до `FIN=1`. Control frames могут вклиниваться между фрагментами, но новый text/binary message нельзя начать, пока предыдущее фрагментированное сообщение не завершено. В одном классическом WS-соединении нет message id и мультиплексирования, поэтому логические `id`, `type` или `stream_id` добавляет само приложение.

## Закрытие, ping/pong и ошибочные HTTP-ответы

В предыдущей секции close, ping и pong уже появились как отдельные `opcode`: `0x8`, `0x9` и `0xA`. Это не data frames, а control frames. Они живут в том же WebSocket-потоке, что text, binary и continuation frames, но подчиняются более строгим правилам.

У control frames есть три важных свойства:

- payload не может быть больше 125 байт;
- control frames не фрагментируются;
- они могут вклиниваться между фрагментами одного data message.

Последний пункт часто путают с мультиплексированием. Если сторона начала фрагментированное text или binary message, она не может до его завершения начать новое text или binary message. Разрешены только control frames между фрагментами:

```text
text FIN=0
ping
continuation FIN=1
```

А вот такая последовательность остаётся протокольной ошибкой:

```text
text FIN=0
binary FIN=1
continuation FIN=1
```

То есть control frames помогают соединению оставаться управляемым даже во время большого сообщения, но не превращают WebSocket в протокол с несколькими параллельными data streams.

### Close frame

Нормальное закрытие на WebSocket-уровне начинается с close frame, то есть control frame с `opcode=0x8`. Это прикладной сигнал протокола WebSocket: сторона сообщает, что больше не будет отправлять data frames и хочет завершить соединение корректно.

Обычная последовательность такая:

1. Одна сторона отправляет close frame.
2. Peer, если сам ещё не отправлял close frame, отвечает своим close frame.
3. После обмена close frame закрывается нижнее соединение: TCP-сокет, а для `wss://` ещё и TLS-сессия поверх него.

```{mermaid}
stateDiagram-v2
    [*] --> Open
    Open --> CloseSent: send close
    Open --> CloseReceived: receive close
    CloseSent --> TcpClose: receive close
    CloseReceived --> TcpClose: send close
    CloseSent --> TcpClose: timeout
    CloseReceived --> TcpClose: timeout
    TcpClose --> [*]
```

Payload close frame может быть пустым. Если он есть, первые 2 байта означают status code, а оставшаяся часть может содержать UTF-8 reason. Для интервью обычно не нужно перечислять каталог close-кодов: важнее понимать саму форму сигнала и то, что reason ограничен маленьким payload control frame.

Close frame нельзя смешивать с TCP FIN или TCP RST. Это разные уровни. WebSocket close frame - байты внутри уже установленного WebSocket-протокола. TCP FIN - закрытие байтового потока транспортным слоем. TCP RST - аварийный сброс соединения. Нормальная WebSocket-библиотека старается провести close handshake, но нижний TCP/TLS всё равно может оборваться без какого-либо WebSocket close frame. Для приложения это обычно выглядит как abnormal closure: соединение исчезло, а аккуратного WS-сигнала закрытия не было.

Практически close handshake не должен висеть бесконечно. Если peer не отвечает close frame или нижний транспорт завис в полуживом состоянии, реализация ставит таймаут и закрывает сокет принудительно. Это не "красивое" завершение, но без такого таймаута серверы и клиенты легко накапливают зависшие соединения.

### Ping и pong

Ping и pong - тоже control frames: `opcode=0x9` и `opcode=0xA`. Их основная роль - проверить liveness соединения и поддержать heartbeat. Это особенно полезно из-за idle timeout у серверов, балансировщиков, прокси и NAT, а также из-за полуживых соединений, где одна сторона уже недоступна, но локальный сокет ещё не получил явную ошибку.

Получив ping, peer должен ответить pong с тем же application data, если соединение ещё не находится в процессе закрытия. Это echo не означает, что приложение обработало предыдущее бизнес-сообщение. Оно означает только, что WebSocket peer на другом конце канала получил ping и смог отправить ответ.

Из этого следуют две практические границы:

- ping/pong подходят для heartbeat и обнаружения "молчаливого" разрыва;
- ping/pong не заменяют application-level ack.

Если приложению нужно знать, что заявка принята, команда выполнена или сообщение записано в базу, это должен подтверждать прикладной протокол в payload: например, `ack`, `request_id`, `status`, `sequence` или другой явный контракт. Pong говорит о живости соединения, а не о бизнес-семантике.

Pong чаще всего является ответом на ping. Протокол также допускает unsolicited pong, то есть pong без предшествующего ping, но для базовой инженерной модели достаточно помнить, что это всё ещё служебный control frame, а не прикладное подтверждение.

В реализации heartbeat обычно превращается в пару таймеров: периодически отправить ping и ожидать matching pong в разумный срок. Если pong не пришёл, соединение закрывают и запускают reconnect/backoff по политике приложения. Конкретные значения таймаутов зависят от среды: локальная сеть, мобильный клиент, корпоративный прокси и биржевой feed требуют разных допусков.

### Если вместо `101` пришёл HTTP-ответ

WebSocket frames начинаются только после успешного opening handshake. Для классического WebSocket поверх HTTP/1.1 ключевой сигнал - ответ `101 Switching Protocols` с корректными WebSocket-заголовками. Если сервер вернул `200 OK`, HTML-страницу, `404 Not Found`, `400 Bad Request`, redirect или любой другой обычный HTTP-ответ, WebSocket-соединение не установлено.

В этой ситуации корректный клиент остаётся на уровне HTTP-ошибки handshake. Он может показать ошибку, обработать redirect согласно своей политике, залогировать тело ответа или закрыть соединение, но он не должен парсить эти байты как WebSocket frames.

Отсюда появляется популярный миф: обычный HTTP/HTML-ответ якобы "нативно" выглядит как WebSocket close. В нём есть зерно байтовой случайности, но нет корректной протокольной логики. Если ошибочная реализация проигнорирует failed handshake и начнёт читать строку вроде:

```http
HTTP/1.1 200 OK
```

как WebSocket frame, первый байт `H` (`0x48`) действительно имеет младшие 4 бита `0x8`, то есть может быть ошибочно принят за close opcode. Но старшие биты того же байта и следующие байты не образуют корректный WebSocket close frame. Такая реализация уже нарушила состояние протокола: она должна была остановиться на том, что handshake не дал `101`. Правильный вывод здесь не "HTTP ответ нормально закрывает WebSocket", а "клиент перешёл к WS-парсеру после failed handshake и получил protocol error".

Эта граница важна в отладке. Если в логах WebSocket-клиента внезапно виден HTML, JSON с HTTP-ошибкой или текстовая страница балансировщика, это почти всегда проблема до WebSocket-фреймов: неверный URL, path, заголовки upgrade, auth, proxy, TLS termination или routing. Искать close frame в таком ответе не нужно.

### Практическая модель для кода и networking

На уровне кода полезно держать несколько независимых состояний:

- WebSocket-протокол открыт и принимает frames;
- close frame отправлен или получен, идёт close handshake;
- heartbeat ждёт pong на последний ping;
- нижний TCP/TLS сокет может закрыться или сломаться независимо от аккуратного WS close.

Эти состояния не стоит сводить к одному флагу `connected`. Соединение может быть формально открыто, но давно не отвечать на ping. Может прийти TCP EOF без close frame. Может быть получен close frame, после которого приложение уже не должно отправлять новые business messages, но ещё должно корректно ответить close frame и закрыть транспорт.

Поэтому рабочая реализация обычно имеет:

- таймаут на opening handshake;
- таймаут на close handshake;
- heartbeat-интервал и timeout ожидания pong;
- лимиты на размеры сообщений и control payload;
- явную обработку EOF, TLS alert, TCP RST и protocol error как разных причин завершения.

Для собеседования не нужно писать полноценный код, но стоит проговорить границу ответственности. Сокет и TLS сообщают о состоянии нижнего канала. WebSocket-парсер сообщает о protocol frames и protocol errors. Приложение решает, как классифицировать завершение: нормальное закрытие, failed handshake, timeout heartbeat, abnormal closure или ошибка собственного прикладного протокола.

### Как это формулировать на интервью

Короткая версия:

> Close, ping и pong в WebSocket - это control frames с opcode `0x8`, `0x9` и `0xA`. Control frames имеют payload не больше 125 байт, не фрагментируются и могут вклиниваться между фрагментами data message. Но новое text или binary message нельзя начать, пока предыдущее фрагментированное сообщение не завершено. Нормальное закрытие начинается close frame: peer обычно отвечает close frame, затем закрывается нижний TCP/TLS-канал. Close frame и TCP FIN/RST - разные уровни. Ping проверяет liveness; на ping отвечают pong с тем же application data, если соединение не закрывается. Ping/pong полезны для heartbeat и idle timeout, но не являются ack бизнес-сообщений. Если opening handshake не дал `101 Switching Protocols`, WebSocket не установлен: `200`, HTML, `404`, `400` или redirect остаются HTTP-ответами, и корректный клиент не парсит их как WS frames.

## HTTP/1.1, HTTP/2, HTTP/3 и место WebSocket

До этого момента статья шла по классическому пути WebSocket: TCP, при `wss://` TLS, затем HTTP/1.1 opening handshake с `Upgrade: websocket`, ответ `101 Switching Protocols` и дальше WebSocket-фреймы. Эта модель до сих пор важна, потому что она самая простая для понимания и самая совместимая в реальной инфраструктуре.

Но рядом с ней давно существуют HTTP/2 и HTTP/3. Они меняют wire-format, модель соединений и мультиплексирование HTTP-запросов, из-за чего легко сделать поспешный вывод: если в HTTP уже появились streams, WebSocket должен был стать ненужным. На практике это не так. HTTP/2 и HTTP/3 развивают транспортную и HTTP-инфраструктуру, а WebSocket остаётся отдельным прикладным контрактом для двунаправленного обмена сообщениями.

### HTTP/1.1: текстовый запрос-ответ и Upgrade

HTTP/1.1 на базовом уровне легко читать глазами: стартовая строка, текстовые заголовки, пустая строка и затем тело сообщения, если оно есть.

```http
GET /image.png HTTP/1.1
Host: example.com
User-Agent: example-client
Accept: image/png
```

Эта простота полезна для отладки, но у HTTP/1.1 слабая модель параллелизма внутри одного соединения. Исторически браузеры и клиенты получали параллелизм через несколько TCP-соединений к одному origin: одно соединение занято одним запросом или ответом, другое может в это время обслуживать следующий ресурс.

В HTTP/1.1 был механизм pipelining: клиент мог отправить несколько запросов подряд, не дожидаясь ответов. Но ответы всё равно должны были приходить в порядке запросов, поэтому медленный первый ответ задерживал последующие. Из-за практических проблем с посредниками и реализациями pipelining не стал тем универсальным решением, которое убрало бы потребность в нескольких TCP-соединениях.

Классический WebSocket использует именно HTTP/1.1-механику Upgrade. Клиент отправляет `GET` с `Upgrade: websocket`, сервер при успехе отвечает `101 Switching Protocols`, и после этого HTTP для данного соединения заканчивается. Дальше те же байты интерпретируются не как HTTP-запросы и ответы, а как WebSocket-фреймы.

```text
HTTP/1.1 request-response
HTTP/1.1 Upgrade
101 Switching Protocols
WebSocket frames
```

Это важная граница: WebSocket не является "ещё одним HTTP/1.1 response body" после `101`. HTTP использовался как способ договориться о переключении протокола, а затем уступил место WebSocket wire-format.

### HTTP/2: frames, streams и одно TCP-соединение

HTTP/2 сохраняет HTTP-семантику: есть методы, пути, статусы, заголовки и тела. Но wire-format уже другой. Вместо текстового потока HTTP/1.1 по соединению идут бинарные HTTP/2 frames.

Каждый HTTP/2 frame имеет служебный заголовок, где есть тип фрейма, флаги, длина payload и `stream_id`. Stream - это логический поток внутри одного HTTP/2-соединения. Запрос за HTML может идти в stream 1, запрос за CSS - в stream 3, запрос за картинку - в stream 5. Физически всё это сериализовано в один TCP byte stream, но HTTP/2-фреймы разных streams могут чередоваться:

```text
HEADERS stream=1
HEADERS stream=3
DATA    stream=1
HEADERS stream=5
DATA    stream=3
DATA    stream=1
DATA    stream=5
```

Именно это обычно имеют в виду под multiplexing в HTTP/2: много логических запросов и ответов живут внутри одного TCP-соединения, а получатель демультиплексирует их по `stream_id`.

Заголовки в HTTP/2 не передаются как обычный текстовый блок. Для них используется HPACK: статическая таблица частых заголовков, динамическая таблица на соединение и компактное кодирование ссылок на уже известные имена и значения. Поэтому расшифрованный TLS-трафик HTTP/2 всё равно не выглядит как читаемый HTTP/1.1: нужно разобрать binary frames и декодировать HPACK header blocks. Тело ответа при этом не становится "другим": JSON, картинка или protobuf остаются payload'ом DATA-фреймов, но их нужно собрать из фреймов нужного `stream_id`.

HTTP/2 stream не равен WebSocket message channel. Stream - это внутренняя единица HTTP/2 для запроса, ответа, фреймов, flow control и состояния протокола. WebSocket - это прикладной двунаправленный message-oriented канал с собственным API и собственными фреймами. То, что оба слова звучат как "поток", не делает их взаимозаменяемыми.

У HTTP/2 остаётся и принципиальное ограничение: он работает поверх одного TCP-соединения. TCP гарантирует доставку байтов в порядке. Если один TCP-сегмент потерян, стек не может отдать приложению более поздние байты до восстановления пропуска. Для HTTP/2 это означает TCP-level head-of-line blocking: потеря на уровне TCP может задержать все HTTP/2 streams этого соединения, даже если логически потерянные данные относились только к одному из них.

### HTTP/3: HTTP-семантика поверх QUIC

HTTP/3 часто неаккуратно называют "QUIC". Корректнее разделять:

- QUIC - транспортный протокол поверх UDP;
- HTTP/3 - HTTP-семантика, отображённая на QUIC streams.

UDP сам по себе не даёт надёжной доставки, порядка байтов, контроля перегрузки или потоков. Но QUIC не использует UDP как "ненадёжный HTTP". Он строит поверх UDP собственный транспортный слой: надёжную доставку, retransmission, multiplexed streams, flow control, congestion control, connection IDs и криптографическое рукопожатие на базе TLS 1.3.

Поэтому фраза "HTTP/3 работает поверх UDP" верна, но неполна. В приложении это не означает "пакеты могут пропадать, и HTTP как-нибудь с этим живёт". Потери обрабатывает QUIC. Разница с TCP в том, что QUIC знает о своих streams и может не блокировать все потоки из-за потери данных одного потока. HTTP/3 тем самым убирает TCP-level head-of-line blocking между QUIC streams.

Это не отменяет всех задержек вообще. Есть общая пропускная способность, congestion control, flow control, приоритеты, лимиты памяти и процессорная цена шифрования. Но исчезает конкретная проблема HTTP/2 поверх TCP: "одна потеря в TCP byte stream задержала байты всех логических HTTP streams".

С точки зрения кода стек выглядит иначе:

```text
HTTP/1.1 or HTTP/2: TCP socket -> TLS library -> HTTP stack
HTTP/3: UDP socket -> QUIC library with TLS 1.3 -> HTTP/3 stack
```

В первом случае приложение обычно отдаёт байты из TCP/TLS в HTTP-парсер. Во втором случае приложение отдаёт UDP datagrams в QUIC-библиотеку, а она уже ведёт соединение, ключи, retransmission, congestion control и QUIC streams.

### Слои в одной картине

```{mermaid}
flowchart TD
    App[Application] --> Http[HTTP semantics]
    App --> Ws[WebSocket message API]

    Http --> H1[HTTP/1.1 text messages]
    Http --> H2[HTTP/2 frames and streams]
    Http --> H3[HTTP/3 frames]

    Ws --> WsH1[H1 Upgrade then WS frames]
    Ws --> ExtConn[Extended CONNECT]

    WsH1 --> H1
    ExtConn --> H2Stream[WebSocket over H2 stream]
    ExtConn --> H3Stream[WebSocket over H3 stream]
    H2Stream --> H2
    H3Stream --> H3

    H1 --> TlsTcp[TLS over TCP when secure]
    H2 --> TlsTcp
    H3 --> Quic[QUIC with TLS 1.3]

    TlsTcp --> Tcp[TCP byte stream]
    Quic --> Udp[UDP datagrams]

    Tcp --> Ip[IP]
    Udp --> Ip
```

Диаграмма намеренно упрощает детали. Её задача - удержать границы: HTTP-семантика может быть закодирована по-разному, HTTP/1.1 и HTTP/2 в защищённом варианте идут через TLS поверх TCP, а HTTP/3 использует QUIC поверх UDP. WebSocket здесь показан не как замена TLS или TCP, а как прикладной message API, который в классическом варианте стартует через HTTP/1.1 Upgrade, а в HTTP/2/3-вариантах отображается на stream через общую идею Extended CONNECT. Конкретные правила включения и поддержки для HTTP/2 и HTTP/3 различаются; это будет темой следующей секции.

### Сравнение HTTP/1.1, HTTP/2 и HTTP/3

| Свойство | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| Базовая семантика | HTTP request/response | HTTP request/response | HTTP request/response |
| Wire-format | Текстовые start line и заголовки | Бинарные frames | HTTP/3 frames поверх QUIC streams |
| Транспорт | Обычно TCP | TCP | QUIC поверх UDP |
| TLS | Для HTTPS/WSS - отдельный TLS-слой поверх TCP | Обычно TLS с ALPN `h2` в браузерном сценарии | TLS 1.3 встроен в QUIC handshake |
| Параллелизм | Исторически несколько TCP-соединений | Много streams в одном TCP-соединении | Много QUIC streams в одном QUIC-соединении |
| Заголовки | Текстовые, без HPACK | HPACK | QPACK |
| Head-of-line blocking | На уровне соединения и порядка ответов в простых моделях | HTTP streams мультиплексируются, но остаётся TCP-level HOL | TCP-level HOL между streams убран, потери обрабатываются на уровне QUIC |
| Классический WebSocket | `GET` + `Upgrade`, затем `101` и WS-фреймы | Не тот же механизм; детали через Extended CONNECT - отдельная тема | Не тот же механизм; детали через Extended CONNECT - отдельная тема |
| Что важно на интервью | Простая читаемая модель, но слабый multiplexing | `stream_id`, binary framing, HPACK, один TCP | HTTP поверх QUIC, UDP не значит "без гарантий" |

### Почему HTTP/2 не убил WebSocket

Идея "HTTP/2 streams заменят WebSocket" звучит правдоподобно, если смотреть только на слово multiplexing. HTTP/1.1 плохо обслуживал много параллельных запросов в одном соединении, HTTP/2 дал streams, значит вроде бы можно больше не держать отдельный realtime-протокол. Но это смешивает разные задачи.

HTTP/2 streams решают мультиплексирование HTTP-запросов и HTTP-ответов. Они полезны, когда браузер грузит HTML, CSS, JavaScript, картинки и API-ответы с одного origin: не нужно на каждый ресурс поднимать отдельный TCP/TLS путь, фреймы разных запросов могут чередоваться в одном соединении, а HTTP-стек диспетчеризует их по `stream_id`.

WebSocket решает другую задачу: дать приложению устойчивый двунаправленный message-oriented канал. У него есть понятный браузерный API, серверные библиотеки, модель сообщений, ping/pong, close handshake и ожидания экосистемы. Приложение думает не в терминах "открыл HTTP/2 stream и пишу туда произвольную смесь фреймов", а в терминах "есть соединение, отправляю и получаю сообщения".

Кроме того, HTTP/2 stream сам по себе не стал универсальным browser API для произвольного двунаправленного обмена. Браузерная и серверная экосистема долго живут с обоими подходами: HTTP/2 и HTTP/3 ускоряют и уплотняют обычный HTTP-трафик, а WebSocket остаётся привычным контрактом для realtime-задач.

Это не значит, что WebSocket навсегда обязан ездить только через HTTP/1.1 Upgrade. Существуют варианты WebSocket поверх HTTP/2 и HTTP/3 через Extended CONNECT, где один WebSocket-канал отображается на HTTP stream. Но это уже тема следующей секции. Для текущей ментальной модели достаточно зафиксировать: HTTP/2/3 развивают транспорт и HTTP-мультиплексирование, но не автоматически заменяют realtime-протокол приложения.

### Как это формулировать на интервью

Короткая версия:

> HTTP/1.1 - текстовый request/response-протокол, который плохо мультиплексирует несколько запросов в одном соединении, поэтому параллелизм исторически получали через несколько TCP-соединений. Классический WebSocket стартует именно через HTTP/1.1 Upgrade: клиент отправляет `GET` с `Upgrade: websocket`, сервер отвечает `101`, и после этого HTTP для этого соединения заканчивается, дальше идут WebSocket-фреймы. HTTP/2 сохраняет HTTP-семантику, но кодирует её бинарными frames и streams в одном TCP-соединении; заголовки сжимаются HPACK, а фреймы демультиплексируются по `stream_id`. При этом HTTP/2 stream - не WebSocket message channel, и TCP-level head-of-line blocking остаётся: потеря TCP-сегмента задерживает все streams этого TCP-соединения. HTTP/3 - это HTTP поверх QUIC, а не синоним QUIC. QUIC работает поверх UDP, но сам реализует надёжность, потоки, flow control, congestion control и использует TLS 1.3. HTTP/3 убирает TCP-level HOL между QUIC streams, но не превращает UDP в "без гарантий". WebSocket не умер с HTTP/2, потому что HTTP streams решают multiplexing HTTP-запросов и ответов, а WebSocket даёт стабильный прикладной двунаправленный канал сообщений с широкой экосистемой.

Следующий уровень детализации - как WebSocket можно запускать не через HTTP/1.1 Upgrade, а поверх HTTP/2 или HTTP/3 через Extended CONNECT.

## WebSocket поверх HTTP/2/3 и Extended CONNECT

Классический WebSocket через HTTP/1.1 `Upgrade` остаётся самым совместимым путём: клиент отправляет `GET`, сервер отвечает `101 Switching Protocols`, и дальше всё соединение перестаёт быть HTTP-соединением и становится WebSocket-каналом.

В HTTP/2 и HTTP/3 такая модель не ложится на протокол напрямую. Там одно нижнее соединение несёт много логических streams. Если бы один stream мог сказать "переключаем всё соединение на WebSocket", он сломал бы соседние streams. Поэтому connection-wide `Upgrade` из HTTP/1.1 для HTTP/2/3 невозможен или, точнее, неуместен: переключать нужно не всё соединение, а один конкретный stream.

Для этого используется Extended CONNECT. Обычный HTTP `CONNECT` создаёт туннель к указанному endpoint'у: после успешного ответа байты внутри туннеля уже не интерпретируются как обычный HTTP request/response. Extended CONNECT развивает эту идею и добавляет pseudo-header `:protocol`, который говорит, какой протокол будет жить внутри stream. Для WebSocket это:

```text
:protocol = websocket
```

Итоговая модель такая: WebSocket-канал живёт внутри одного HTTP/2 или HTTP/3 stream. Снаружи соединение остаётся HTTP/2 или HTTP/3 и может одновременно обслуживать другие streams, а внутри выбранного stream после успешного Extended CONNECT стороны используют обычный WebSocket Protocol.

### Как выглядит запрос

Упрощённый HTTP/2-запрос для WebSocket через Extended CONNECT можно представить так:

```text
:method = CONNECT
:protocol = websocket
:scheme = https
:path = /chat
:authority = example.com
sec-websocket-version = 13
sec-websocket-protocol = chat
```

Успешный ответ - не `101`, а обычный успешный статус для CONNECT:

```text
:status = 200
sec-websocket-protocol = chat
```

`sec-websocket-protocol` в ответе нужен только если сервер выбрал один из предложенных subprotocol. Если subprotocol не договаривался или сервер ничего не выбрал, этого заголовка может не быть.

После успешного ответа peers передают WebSocket frames внутри этого stream примерно так, как если бы stream был нижним TCP-соединением для WebSocket. Важно слово "примерно": нижний транспорт уже не отдельный TCP byte stream, а поток внутри HTTP/2 или HTTP/3, но WebSocket wire-format выше остаётся тем же: фреймы, opcode, payload, ping/pong, close и правила маскирования.

### Чем это отличается от HTTP/1.1 opening handshake

В классическом HTTP/1.1 handshake клиент отправляет:

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: ...
Sec-WebSocket-Version: 13
```

Сервер при успехе отвечает:

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: ...
```

В HTTP/2-варианте по RFC 8441 важные отличия такие:

- нет `GET` + `Upgrade`;
- нет ответа `101 Switching Protocols`;
- нет `Connection: Upgrade` и `Upgrade: websocket`;
- `Host` передаётся как pseudo-header `:authority`;
- `Sec-WebSocket-Key` и `Sec-WebSocket-Accept` не используются в этом пути;
- роль явного "это WebSocket" выполняет `:protocol = websocket`, а корректность режима контролируется HTTP/2 stack, настройками соединения и правилами Extended CONNECT.

Отсутствие `Sec-WebSocket-Key` и `Sec-WebSocket-Accept` не означает, что WebSocket стал менее защищённым. Эти заголовки в HTTP/1.1 никогда не были криптографической аутентификацией сервера. В `wss://` настоящую защиту даёт TLS, а в HTTP/2 path проверка "мы действительно открываем WebSocket, а не случайно получили HTML-страницу" переехала в другой механизм: pseudo-header `:protocol`, negotiated support и состояние HTTP/2 stream.

### HTTP/2: один TCP, много streams

Для HTTP/2 сервер должен заранее объявить поддержку Extended CONNECT через setting:

```text
SETTINGS_ENABLE_CONNECT_PROTOCOL = 1
```

Без этого клиент не должен считать, что `CONNECT` с `:protocol = websocket` доступен. Иначе он рискует отправить запрос, который другая сторона или промежуточный компонент не поймёт.

Если поддержка есть у клиента, сервера и посредников, один HTTP/2 connection может нести несколько WebSocket-каналов как разные streams:

```text
TCP + TLS + HTTP/2 connection
  stream 1: обычный HTTP request/response
  stream 3: WebSocket /chat
  stream 5: WebSocket /prices
  stream 7: обычный HTTP request/response
```

Это и есть аккуратный ответ на вопрос "можно ли посадить несколько WebSocket на один TCP". Да, но только если это WebSocket-over-H2, где каждый WebSocket отображён на отдельный HTTP/2 stream. Классический RFC 6455 Upgrade поверх HTTP/1.1 занимает один TCP connection одним WebSocket-каналом: после `101` на этом TCP уже нет места для других HTTP-запросов или других WebSocket-каналов на уровне протокола.

Ограничение HTTP/2 при этом остаётся прежним: физически всё едет через один TCP byte stream. HTTP/2 умеет мультиплексировать streams на своём уровне, но потеря TCP-сегмента может задержать доставку байтов для всех streams этого TCP-соединения.

### HTTP/3: та же идея поверх QUIC streams

Для HTTP/3 модель похожа по HTTP-семантике, но не стоит говорить, что это "тот же HTTP/2". HTTP/3 переносит идею Extended CONNECT на QUIC streams. Нижнее соединение здесь не TCP, а QUIC connection поверх UDP, внутри которого есть независимые QUIC streams.

WebSocket over HTTP/3 описан в RFC 9220. Запрос использует тот же смысл pseudo-headers:

```text
:method = CONNECT
:protocol = websocket
:scheme = https
:path = /chat
:authority = example.com
```

Setting для включения Extended CONNECT семантически такой же: сервер объявляет, что CONNECT с `:protocol` поддерживается. Но для HTTP/3 это отдельная зарегистрированная настройка, а не буквальное переиспользование HTTP/2 setting на другом wire-format.

Плюс HTTP/3 в этой картине - отсутствие TCP-level head-of-line blocking между streams. Потеря данных, относящихся к одному QUIC stream, не обязана блокировать доставку данных других QUIC streams так, как это происходит при потере байта в одном TCP-потоке под HTTP/2. Это не отменяет общую перегрузку, flow control, лимиты памяти и поддержку со стороны инфраструктуры, но убирает конкретный TCP-level HOL.

Закрытие stream в HTTP/3 для этой модели соответствует закрытию нижнего соединения в классическом RFC 6455 смысле: если stream завершён или сброшен, WebSocket-канал больше не может продолжать обмен фреймами. Если endpoint не поддерживает указанный `:protocol`, он может ответить ошибкой, например `501 Not Implemented`, вместо открытия WebSocket stream.

Тонкое различие в формулировке: в HTTP/3 это уже не "несколько WebSocket на одном TCP". TCP здесь нет. Корректнее говорить: несколько WebSocket-каналов могут жить как разные streams в одном QUIC connection поверх UDP, если весь тракт поддерживает WebSocket over HTTP/3.

### Fallback на практике

Практический flow установления WebSocket-соединения начинается с WebSocket URI и набора возможных нижележащих HTTP-путей. Для `wss://example.com/chat` клиент знает схему `wss`, host, path и порт `443` по умолчанию. Дальше появляется важная развилка транспортного уровня: HTTP/1.1 и HTTP/2 используют TCP/TLS, а HTTP/3 использует QUIC поверх UDP. Из-за этой развилки схема выбора и fallback может распадаться на несколько connection candidates.

На уровне транспорта это две разные цепочки:

```{mermaid}
flowchart LR
    WSS["wss://example.com/chat"] --> TCP["TCP/443"]
    TCP --> TLS["TLS handshake"]
    TLS --> H2H1["ALPN: h2 или http/1.1"]

    WSS --> UDP["UDP/443"]
    UDP --> QUIC["QUIC handshake<br/>with TLS 1.3"]
    QUIC --> H3["ALPN: h3"]
```

В этой модели `ALPN` работает внутри конкретного handshake. В TCP/TLS path он выбирает между `h2` и `http/1.1`; в QUIC/TLS path он подтверждает `h3`. Поэтому полезно разделять два вопроса: как клиент получает транспортные кандидаты и что потом выбирается внутри каждого кандидата.

TCP/TLS-кандидат есть у клиента без дополнительной разметки. Для `wss://example.com/chat` клиент может открыть TCP-соединение к `example.com:443`, выполнить TLS handshake с ALPN `h2, http/1.1` и получить от сервера выбранный HTTP-протокол. Если выбран `http/1.1`, дальше возможен классический WebSocket Upgrade. Если выбран `h2`, WebSocket возможен через Extended CONNECT, если сервер и весь маршрут объявляют и поддерживают этот режим.

QUIC/HTTP/3-кандидат появляется из отдельных discovery-механизмов:

- DNS `HTTPS`/`SVCB` record даёт сигнал до первого HTTP-ответа. Для WebSocket это применимо потому, что bootstrap всё равно использует HTTP connection machinery: RFC 9460 относит HTTP-based протоколы, включая WebSocket, к сценариям, где клиент может использовать `HTTPS` records. DNS может вернуть service binding с `alpn="h3"` и, при необходимости, alternative endpoint или параметры подключения. Тогда клиент уже на cold start может построить UDP/QUIC-кандидат, выполнить QUIC handshake и подтвердить `h3` через ALPN внутри QUIC/TLS.
- `Alt-Svc` header появляется после ответа по уже доступному HTTP-пути, например после первого запроса через HTTP/2 или HTTP/1.1 к тому же origin. Ответ вида `Alt-Svc: h3=":443"; ma=86400` говорит клиенту, что тот же origin доступен через HTTP/3 на указанном endpoint'е. Клиент кэширует это знание на время `ma`, и следующие подключения к этому origin уже могут начинаться с QUIC-кандидата. `Alt-Svc` не меняет WebSocket URI и не переносит authority без проверки: клиент всё равно проверяет, что alternative service авторитетен для исходного origin.
- Существующая сессия или connection pool сокращают выбор. Если у клиента уже есть активное QUIC-соединение к этому origin, следующий WebSocket-over-H3 можно пытаться открыть как новый stream в нём. Если есть пригодное HTTP/2-соединение, WebSocket-over-H2 аналогично начинается с нового Extended CONNECT stream.
- Кэш неуспешного QUIC тоже влияет на выбор. Если клиент недавно видел, что UDP заблокирован, QUIC handshake не отвечает или HTTP/3 для origin помечен как broken, он может сразу идти TCP/TLS path и не тратить задержку на заведомо плохой UDP-кандидат.
- Racing - это стратегия конкретной реализации, а не отдельный протокол. При наличии свежего `Alt-Svc` или DNS `HTTPS/SVCB` с `h3` клиент может запустить конкурирующие connection jobs: TCP/TLS как базовый путь и QUIC/UDP как alternative path. Дальше используется тот путь, который успешно установился и подходит политике клиента; если QUIC не установился, остаётся TCP path.

Один и тот же WebSocket endpoint при этом может быть доступен одновременно через `TCP/443` для HTTP/1.1 или HTTP/2 и через `UDP/443` для HTTP/3: это разные transport endpoints, они не конфликтуют.

Схематически discovery-часть выглядит так:

```{mermaid}
flowchart TD
    URI["wss://example.com/chat"] --> Pool{"Есть пригодный<br/>H2/H3 connection?"}
    Pool -->|Да| Reuse["Открыть новый<br/>Extended CONNECT stream"]
    Pool -->|Нет| Build["Построить connection candidates"]

    Build --> TCP["Базовый TCP/TLS candidate<br/>ALPN: h2, http/1.1"]
    Build --> DNS["DNS HTTPS/SVCB<br/>с h3"]
    Build --> AltSvc["Свежий Alt-Svc<br/>с h3 в кэше"]

    DNS --> H3Cold["QUIC/UDP candidate<br/>до первого HTTP-ответа"]
    AltSvc --> H3Next["QUIC/UDP candidate<br/>для следующего подключения"]
    H3Cold --> H3Candidate["QUIC/UDP candidate"]
    H3Next --> H3Candidate
    H3Candidate --> Broken{"UDP/H3 недавно<br/>ломался для origin?"}
    Broken -->|Да| Lower["Пропустить или понизить<br/>приоритет QUIC"]
    Broken -->|Нет| H3["QUIC/TLS ALPN: h3"]

    TCP --> Strategy{"Connection strategy"}
    H3 --> Strategy
    Lower --> Strategy

    Strategy -->|Только TCP path| TCPConn["TCP/TLS connection"]
    Strategy -->|QUIC path выбран| QUICConn["QUIC connection"]
    Strategy -->|Реализация запускает racing| Race["Конкурирующие TCP и QUIC jobs"]
    Race --> Winner{"Какой path успешно<br/>установился и принят политикой?"}
    Winner -->|TCP| TCPConn
    Winner -->|QUIC| QUICConn
```

После выбора HTTP-версии начинается WebSocket-specific развилка. Если соединение установлено как `http/1.1`, клиент может выполнить классический `GET` с `Upgrade: websocket` в этом HTTP/1.1 connection. Если выбран `h2` или `h3`, WebSocket открывается не через connection-wide Upgrade, а через Extended CONNECT внутри отдельного stream. Для этого должны совпасть несколько условий: выбранный протокол поддерживается клиентской библиотекой, сервер объявил нужную поддержку через settings, а промежуточная инфраструктура не ломает такой путь.

Если HTTP/2/HTTP/3 Extended CONNECT недоступен или запрос неуспешен, fallback на HTTP/1.1 требует нового connection. Нельзя внутри уже выбранного `h2` или `h3` соединения внезапно отправить HTTP/1.1 `GET` с `Upgrade: websocket`: wire-format и состояние соединения уже другие. Клиент должен открыть отдельный TCP/TLS path с такой negotiation или политикой, где будет выбран `http/1.1`, и уже там выполнить классический opening handshake.

На реальной системе итог зависит не только от браузера или клиента. Важны сервер, reverse proxy, CDN, load balancer, корпоративные прокси, настройки TLS/ALPN, доступность UDP и конкретная библиотека.

После discovery можно свернуть WebSocket-specific часть до отдельной схемы:

```{mermaid}
flowchart TD
    Path["Выбранный HTTP path"] --> Proto{"HTTP protocol"}

    Proto -->|http/1.1| H1["GET /chat<br/>Upgrade: websocket"]
    H1 --> H1101{"101 Switching Protocols?"}
    H1101 -->|Да| Classic["WS frames<br/>on connection"]
    H1101 -->|Нет| Fail["Opening handshake failed"]

    Proto -->|h2 или h3| Ext{"Extended CONNECT<br/>доступен?"}
    Ext -->|Да| Connect["CONNECT<br/>:protocol = websocket"]
    Connect --> Status{"Status 200?"}
    Status -->|Да| Stream["WS frames<br/>on stream"]
    Status -->|Нет| NewH1["Открыть новый<br/>HTTP/1.1 connection"]
    Ext -->|Нет| NewH1
    NewH1 --> H1
```

### Как это формулировать на интервью

Короткая версия:

> Классический WebSocket начинается с URI `ws://` или `wss://`; для защищённого варианта `wss://` клиент сначала устанавливает TCP и TLS, а затем отправляет HTTP/1.1 `GET` + `Upgrade`. При успехе сервер отвечает `101 Switching Protocols`, и дальше по тому же соединению идут WebSocket frames. В HTTP/2 и HTTP/3 connection-wide Upgrade не подходит, потому что одно нижнее соединение несёт много streams. Для этого есть Extended CONNECT: запрос идёт как `:method = CONNECT` с `:protocol = websocket`; при успехе приходит `:status = 200`, и дальше WebSocket frames идут внутри одного H2/H3 stream. В HTTP/2 сервер должен объявить `SETTINGS_ENABLE_CONNECT_PROTOCOL = 1`; в HTTP/3 используется отдельная настройка для той же идеи поверх QUIC streams. В практическом flow сначала строятся транспортные кандидаты. TCP/TLS path доступен без отдельной разметки и через ALPN выбирает `h2` или `http/1.1`. QUIC/UDP path появляется либо заранее из DNS `HTTPS/SVCB` с `h3`, либо из кэша `Alt-Svc`, который был получен после предыдущего HTTP-ответа, либо из уже открытой QUIC-сессии. Некоторые клиенты при наличии свежего H3-сигнала запускают TCP и QUIC как конкурирующие connection jobs; если QUIC не отвечает или origin помечен как broken for QUIC, используется TCP/TLS path. TCP/443 и UDP/443 могут одновременно обслуживать один WebSocket endpoint. Если уже выбран `http/1.1`, Upgrade можно делать в этом соединении; если уже выбран `h2`/`h3` и Extended CONNECT недоступен, fallback на HTTP/1.1 требует нового connection.

## Как это выглядит в коде

Все предыдущие секции можно свести к одному практическому правилу: библиотека может спрятать детали, но не отменяет слои. Код всё равно где-то проходит через адресацию, сокет, TLS или QUIC, HTTP-стек, WebSocket handshake, фрейминг и прикладные callbacks.

Если пользоваться готовым WebSocket-клиентом, большая часть этой цепочки будет внутри библиотеки. Но на интервью обычно проверяют не знание конкретного API, а понимание, где появляются partial reads, backpressure, таймауты, TLS-проверки, HTTP-статусы, WebSocket close и reconnect.

### Классический `wss://` поверх HTTP/1.1

Низкоуровневая модель для классического WebSocket выглядит так:

```text
getaddrinfo
-> socket
-> connect
-> TLS handshake
-> HTTP/1.1 Upgrade
-> WebSocket parser
-> application messages
```

До `connect()` приложение работает с именем, портом и адресами. После `connect()` есть TCP byte stream. После TLS handshake прикладные байты уже нельзя корректно читать и писать напрямую через `recv()`/`send()`, если соединение защищённое: данные должны проходить через TLS API, например `SSL_read()` и `SSL_write()` в OpenSSL-подобной модели.

Поверх TLS приложение отправляет HTTP/1.1 opening handshake:

```text
GET /feed HTTP/1.1
Upgrade: websocket
Connection: Upgrade
...
```

Если ответ дал корректный `101 Switching Protocols`, дальше HTTP-парсер для этого соединения больше не нужен: те же bytes из TLS-слоя переходят в WebSocket-парсер. Он уже читает `FIN`, `opcode`, длины, masking key, payload, собирает сообщения из фрагментов и обрабатывает control frames.

Важный практический вывод: `SSL_read()` может вернуть не "одно WebSocket-сообщение", а произвольный кусок защищённого byte stream. WebSocket-библиотека или ваш код должны уметь накопить буфер и разобрать из него фреймы инкрементально.

### Неблокирующий режим и state machines

В production сетевой код часто работает в неблокирующем режиме или через event loop. Тогда вся цепочка превращается в набор state machines:

- TCP connect может завершиться позже;
- TLS handshake может потребовать ещё чтения или записи;
- `SSL_write()` может принять не все прикладные данные;
- WebSocket parser может ждать недостающие байты заголовка или payload;
- HTTP/2 или QUIC stack может остановить отправку из-за flow control;
- приложение может хотеть отправлять быстрее, чем сеть или peer способны принимать.

Поэтому один флаг `connected` почти всегда слишком грубый. Практически нужны отдельные состояния: TCP connected, TLS handshake done, HTTP Upgrade done, WebSocket open, close sent, waiting for pong, reconnect scheduled, output queue has backpressure.

Backpressure означает, что данные нельзя бесконечно складывать в память только потому, что приложение хочет отправлять ещё. Если нижний слой не принимает запись, верхний слой должен остановить чтение из upstream, ограничить очередь, сбросить медленного клиента или применить политику деградации. Это одинаково важно для классического WebSocket, HTTP/2 streams и QUIC streams.

### HTTP/2 в стиле nghttp2

В HTTP/2 приложение обычно не получает "целый HTTP-ответ" из TCP. На проводе есть один TCP byte stream, в котором сериализованы HTTP/2 frames разных streams. TCP или TLS могут вернуть половину H2-frame, несколько frames сразу или кусок DATA-frame.

Модель библиотеки уровня nghttp2 такая:

1. приложение читает bytes из TCP/TLS;
2. передаёт bytes в HTTP/2 session object;
3. библиотека инкрементально восстанавливает H2 frames;
4. по `stream_id`, типу frame и flags обновляет внутреннее состояние;
5. вызывает callbacks: пришли заголовки, пришёл кусок DATA, stream завершён, произошла ошибка.

Контекст запроса или ответа обычно хранит приложение, например в таблице по `stream_id`:

```text
stream_id -> RequestContext
stream_id -> ResponseBuffer
stream_id -> WebSocketChannel
```

Если это WebSocket over HTTP/2, то один H2 stream становится носителем одного WebSocket-канала. H2-библиотека демультиплексирует bytes по `stream_id`, а WebSocket-слой внутри конкретного stream уже разбирает WebSocket frames. Это два разных уровня фрейминга: HTTP/2 frames снаружи и WebSocket frames внутри payload выбранного stream.

На отправку логика симметрична. Приложение не обязано само раскладывать каждый байт HTTP/2 на wire. Оно подаёт библиотеке намерение: открыть stream, отправить заголовки, отправить DATA, закрыть stream. Библиотека формирует frames, учитывает connection-level и stream-level flow control, максимальный размер frame и очередность отправки. Но приложение всё равно должно реагировать на backpressure и не считать, что callback "можно писать" означает "можно положить в память бесконечный объём".

### HTTP/3 и QUIC

Для HTTP/3 стек выглядит иначе. Здесь нет TCP-сокета, поверх которого лежит TLS-библиотека в привычном TLS-over-TCP виде. Обычно приложение имеет UDP socket и QUIC-библиотеку. QUIC-библиотека внутри ведёт TLS 1.3 handshake, ключи, retransmission, congestion control, flow control, connection IDs и streams.

Упрощённо:

```text
UDP socket
-> QUIC library
-> HTTP/3 stack
-> HTTP streams or WebSocket over H3 stream
-> application callbacks
```

Это не значит, что приложение вручную чинит потерянные UDP datagrams. Потери, повторную передачу и порядок bytes внутри stream обеспечивает QUIC. Но event loop меняется: вместо "прочитал bytes из TCP и отдал в TLS/HTTP parser" приложение получает UDP datagrams, отдаёт их QUIC stack, а тот уже сообщает, какие stream events произошли и какие datagrams нужно отправить наружу.

Для разработчика полезная граница такая: QUIC-библиотека ближе по ответственности к сочетанию TCP + TLS + multiplexing, чем к простой обёртке над UDP. HTTP/3 stack строится поверх QUIC streams, а WebSocket over HTTP/3, если поддержан, живёт внутри одного такого stream через Extended CONNECT.

### Слои и callbacks

```{mermaid}
flowchart TD
    App[Application logic] --> Api[Protocol API callbacks]

    Api --> WsH1[WS over HTTP/1.1]
    Api --> H2[HTTP/2 stack]
    Api --> H3[HTTP/3 stack]

    WsH1 --> H1[HTTP/1.1 Upgrade]
    H2 --> H2Streams[H2 streams]
    H3 --> H3Streams[H3 streams]
    H1 --> Tls[TLS library]
    H2Streams --> Tls
    H3Streams --> Quic[QUIC library]

    Tls --> Tcp[TCP socket]
    Quic --> Udp[UDP socket]

    Tcp --> Kernel[Kernel and network]
    Udp --> Kernel
```

Схема не говорит, что в реальном коде обязательно будут отдельные объекты с такими именами. Многие библиотеки объединяют несколько слоёв: WebSocket-клиент может внутри сам делать TLS и HTTP Upgrade, HTTP/3-библиотека может включать QUIC, а фрейминг может быть спрятан за callback'ами. Если используется WebSocket поверх HTTP/2 или HTTP/3, WebSocket-канал будет жить внутри соответствующего stream, как в предыдущей секции. Но ответственность слоёв остаётся полезной: если ломается TLS verification, это не ошибка WebSocket parser; если пришёл не `101`, это failed HTTP Upgrade; если приложение не отвечает на ping, это не проблема DNS.

### Практические ограничения

На собеседовании хорошо звучит не список библиотек, а список мест, где реальная система ломается:

- file descriptors ограничивают количество открытых sockets;
- ephemeral ports ограничивают число одновременных исходящих TCP-соединений к одному направлению и особенно заметны при частых reconnect;
- `TIME_WAIT` и NAT state могут мешать агрессивному переподключению;
- TLS handshake дорогой по RTT и CPU, поэтому reconnect/backoff важны;
- proxy, load balancer и NAT часто имеют idle timeout, поэтому нужны heartbeat и ping/pong;
- WebSocket frame/message size limits защищают память;
- HTTP/2 и QUIC flow control могут остановить stream, даже если приложение хочет писать дальше;
- выходные очереди нужно ограничивать, иначе медленный peer превращается в рост памяти;
- partial read/write - нормальное состояние, а не исключение.

Для классического WebSocket ещё важно помнить: если нужно много логических запросов внутри одного соединения, WebSocket сам не добавит `stream_id`. Это поле должно быть в payload, например `request_id`, `channel`, `type` или `sequence`. Если вместо этого открывать много WebSocket-соединений, появляются другие лимиты: file descriptors, TLS state, лимиты сервера, прокси, NAT и ephemeral ports.

### Как это формулировать на интервью

Короткая версия:

> В коде я мыслю WebSocket не как один magic socket, а как стек. Для классического `wss://`: `getaddrinfo` даёт адреса, `socket/connect` создаёт TCP byte stream, TLS-библиотека делает handshake и дальше все прикладные bytes идут через `SSL_read`/`SSL_write`, затем HTTP/1.1 Upgrade даёт `101`, после чего WebSocket parser разбирает frames и собирает messages. В неблокирующем режиме TCP, TLS, HTTP/WS parser и приложение - это state machines с partial reads/writes и backpressure. Для HTTP/2 библиотека вроде nghttp2 получает bytes из TCP/TLS, парсит H2 frames, демультиплексирует по `stream_id` и вызывает callbacks; приложение хранит контекст по stream id. Для HTTP/3 обычно есть UDP socket и QUIC-библиотека, которая внутри ведёт TLS 1.3, retransmission, congestion control и streams, а HTTP/3 работает поверх этих streams. Практически нужно помнить про fd limits, ephemeral ports, TIME_WAIT, NAT/proxy idle timeouts, heartbeat, reconnect/backoff, лимиты буферов, размеры сообщений и flow control.
