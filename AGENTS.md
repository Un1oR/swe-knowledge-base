# Repository Guidelines

## Project Structure & Module Organization

Этот репозиторий используется как база знаний по программной инженерии для подготовки к собеседованиям.

Сейчас в дереве репозитория важно следующее:

- `raw/chatgpt/` содержит исходные экспорты чатов и считается слоем исходных материалов;
- `docs/` предназначен для публикуемых статей в формате MyST.

## Documentation Workflow

Репозиторий пополняется двумя путями:

1. Новые материалы по темам программной инженерии пишутся вместе с агентом напрямую в `docs/`.
2. Уже существующие материалы из ChatGPT импортируются через локальный скилл `chatgpt-import-article`.

Если импортный workflow требует промежуточных артефактов, их структуру определяет сам скилл.

## Writing Conventions

Репозиторные документы, спецификации и публикуемые статьи писать на русском языке, если пользователь явно не просит другое.

Финальные статьи должны:

- использовать MyST-нотацию;
- быть самостоятельными статьями, а не видимыми расшифровками формата вопрос-ответ;
- использовать Mermaid-диаграммы по умолчанию там, где они действительно проясняют материал;
- подчёркивать скользкие или часто неверно понимаемые места только тогда, когда это улучшает ясность текста.

## Persistent Docs And Skills

1. `AGENTS.md` хранит только repo-level договорённости: назначение репозитория, язык, роли основных каталогов и общие требования к публикуемым материалам.
2. Если для конкретного workflow есть специализированный скилл, его внутренний процесс не дублируется в `AGENTS.md`. `AGENTS.md` может только описать наличие такого режима работы и сослаться на соответствующий скилл.
3. В постоянные документы попадают только принятые решения, действующие правила и полезные рабочие инструкции. Отброшенные варианты, промежуточные ветви рассуждений, локальные анти-примеры и случайные защитные запреты туда не переносятся.
4. Скиллы не должны симулировать или дублировать логику harness. В них не нужно писать boilerplate вроде “сначала перечитай `AGENTS.md`”, “если локальные правила конфликтуют...” и аналогичные общесистемные оговорки.
5. Repo-level документы должны быть стабильными и долговечными. Если документ временный, exploratory или допускает удаление, на него не должны опираться `AGENTS.md` и рабочие скиллы.
6. Предпочитать позитивные нормы defensive prose: фиксировать желаемый результат, границы ответственности и ожидаемые артефакты, а не коллекционировать запреты и случайные антипаттерны.
7. Общие правила не дублировать в task-specific skill'ах без необходимости. Если правило уже обеспечивается системой, harness'ом или отдельным специализированным скиллом, не копировать его в каждый новый скилл.
